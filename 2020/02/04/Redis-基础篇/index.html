Redis-基础篇
发表于 2020-02-04| 更新于 2020-03-04|RedisRedis-基础篇

为什么用Redis

Redis 关键的是查询数据用内存中的缓存，这样不用去查数据库，所以就带来了两方面的优点：

  • 高性能

    把复杂的查询的数据结果放在 redis 中，查询走 redis，速度更快。

  • 高并发

    如果所有请求都走数据库，数据库支持的并发量有限，而缓存能支持更高并发量。

数据类型

我们使用 Redis 一般都是 key-value 的形式：

  • key 都是 string 类型；
  • value 有 string、hash、list、set、sortset，还有 geo 等。

Redis 对 key-value 封装成对象，key 是一个对象，value 也是一个对象。每个对象都有 type (类型)、encoding (编码)、ptr (指向底层数据结构的指针)来表示。

我们用 Redis 创建键值对的时候，会创建两个对象，key 的对象就是 string 类型，value 会创建对应的数据类型的对象。各种数据类型和对应的底层数据结构关系如下图。

下面就来介绍一下这些数据结构。

数据结构

SDS简单动态字符串

Redis 的 String 字符串（simple dynamic string，SDS）类型和 C 语言的字符串不一样，数据结构如下：

  • 空闲长度；
  • 已用长度；
  • 存放字符的缓冲区（C 的字符串就只有这个缓冲区）。

SDS与C的字符串表示比较

SDS 相比于 C 语言的 String 的优点：

  1. 计算字符串长度，SDS 只需要获取 len 属性，O(1) 复杂度；原生的需要遍历，O(1) 复杂度；

  2. SDS 动态扩容，杜绝缓冲区溢出。SDS 修改时，先会判断空间够不够，不够的话会自动进行缓冲区扩容；不会出现原生那种缓冲区空间不够的情况；

  3. 减少修改字符串带来的内存重分配。

    • 原生的字符串执行 append 之前需要先内存重新分配一个大的空间，不然的话就会有内存溢出的问题，如果截断字符串，也需要内存重新分配一个小的空间，不然的话就会有内存泄漏的问题。

    • SDS 通过空间预分配解决字符串扩大的问题；SDS 通过惰性空间释放解决字符串缩小的问题。

  4. 二进制安全。原生字符串因为使用空字符作为结尾标识，所以只能存字符串；SDS 可以存图片、文件等。

链表

  • 双向链表。

字典/哈希表

  • 字典里封装了哈希表，和 list 中封装双向链表一样；
  • 哈希冲突解决也是用链地址法，但 Redis 是加到链表头部，Java 1.8 是加到链表尾部；
  • 字典中是会有两个哈希表的，一个是存数据的，一个是扩容时候用的，这种方式称为渐进式扩容，防止哈希表中数据量很大，一下子扩容这个哈希表不能用；

跳跃表

跳跃表原理

核心是对有序链表中的部分节点进行拔高，再建立一层索引链表。当数据量多的时候，可以建立多层索引链表，这样查询的时候先比较索引链表，然后在符合区间内下沉链表，最后回到原始链表中查询。

插入节点

插入节点的时候，需要解决一个问题：什么时候决定拔高一层索引，插入节点要不要建高层索引？

解决这个问题，有一个很简单的方法：抛硬币。

当我们插入一个节点的时候，会做如下两步操作：

  1. 根据索引找到插入元素的位置，在原始链表中插入数据，时间复杂度 O(logn)；
  2. 用抛硬币的方式，来决定插入元素需不需要拔高，如果结果为正，则继续抛，继续拔高，当结果出现反的时候停止；
删除节点

删除节点很简单，找到节点之后，把原始链表和其上层的所有节点都删除。

Redis中跳跃表实现

  • Redis 中的跳跃表由两部分组成，一个保存跳跃表的一些信息（表头节点、表尾节点、长度），可以加快访问；另一个是跳跃表的节点；
  • Redis 决定跳跃表节点的层数是通过随机一个 1 到 32 之间的随机数决定的（不是抛硬币）；
  • 跳跃表中节点先按照分值大小排序，分值相同的话，再按照节点对象大小排序。

整数集合

  • 当一个set(集合)只包含整数值元素，并且元素的数量不多时，Redis就会采用整数集合(intset)作为set(集合)的底层实现；

  • 整数集合(intset)保证了元素是不会出现重复的，并且是有序的(从小到大排序)。

压缩列表

  • 压缩列表是 Redis 为了节约内存而开发的， 由一系列特殊编码的连续内存块组成的顺序型数据结构；
  • 节点存放的是前驱的指针，所以需要从尾到头遍历。

数据类型和数据结构选择

我们在看下数据类型和数据结构对应关系的这张表。

String

String 有三种编码类型：int、embstr、row：

  • int：整数值，这个整数值可以使用long类型来表示

    • 如果是浮点数，那就用embstr或者raw编码。具体用哪个就看这个数的长度了

  • embstr：字符串值，这个字符串值的长度小于32字节

  • raw：字符串值，这个字符串值的长度大于32字节

embstr和raw的区别：

  • raw分配内存和释放内存的次数是两次，embstr是一次
  • embstr编码的数据保存在一块连续的内存里面

编码之间的转换：

  • int类型如果存的不再是一个整数值，则会从int转成raw
  • embstr是只读的，在修改的时候回从embstr转成raw

embstr 编码是专门用于保存短字符串的一种优化编码方式，raw 分配内存的时候需要分别为 redisObject 和 sdshdr 分配两次内存，而 embstr 中 redisObject 和 sdshdr 这两个是连续的内存块，一次分配就可以了。

List

  • ziplist：字符串元素的长度都小于64个字节&&总数量少于512个
  • linkedlist：字符串元素的长度大于64个字节||总数量大于512个

编码之间的转换：

  • 原本是ziplist编码的，如果保存的数据长度太大或者元素数量过多，会转换成linkedlist编码的

hash

  • ziplist：key和value的字符串长度都小于64字节&&键值对总数量小于512
  • hashtable：key和value的字符串长度大于64字节||键值对总数量大于512

编码之间的转换：

  • 原本是ziplist编码的，如果保存的数据长度太大或者元素数量过多，会转换成hashtable编码的

set

  • intset：保存的元素全都是整数&&总数量小于512
  • hashtable：保存的元素不是整数||总数量大于512

编码之间的转换：

  • 原本是intset编码的，如果保存的数据不是整数值或者元素数量大于512，会转换成hashtable编码的

sortset

  • ziplist：元素长度小于64&&总数量小于128
  • skiplist：元素长度大于64||总数量大于128

skiplist 编码同时采用跳跃表和哈希表来实现，节点数据会同时保存在跳跃表和哈希表中，这样综合了两个的优势：

  • 哈希表根据 key 查找 O(1) 的复杂度，但哈希表是无序的，如果是范围操作，性能差；
  • 跳跃表适合范围操作，但根据 key 查找时间复杂度 O(logN)。

skiplist 的有序集合能够达到插入的时间复杂度为 O(logn)，根据成员查分值的时间复杂度为 O(1)。

编码之间的转换：

  • 原本是ziplist编码的，如果保存的数据长度大于64或者元素数量大于128，会转换成skiplist编码的。

数据类型的使用场景

  • string–>简单的key-value
  • list–>有序列表(底层是双向链表)–>可做简单队列
  • set–>无序列表(去重)–>提供一系列的交集、并集、差集的命令
  • hash–>哈希表–>存储结构化数据
  • sortset–>有序集合映射(member-score)–>排行榜

单机数据库

数据库的数据结构

Redis 的数据库就是使用字典(哈希表)来作为底层实现的，对数据库的增删改查都是构建在字典(哈希表)的操作之上的。

键的过期时间

因为我们的内存是有限的。所以我们会干掉不常用的数据，保留常用的数据。这就需要我们设置一下键的过期(生存)时间了。

过期策略

Redis采用的是惰性删除+定期删除两种策略，所以说，在Redis里边如果过期键到了过期的时间了，未必被立马删除的！

  • 惰性删除

  • 每次从键空间取键的时候，判断一下该键是否过期了，如果过期了就删除。

  • 定期删除

  • 每隔一段时间去删除过期键。

内存淘汰机制

如果定期删除漏掉了很多过期key，也没及时去查(没走惰性删除)，大量过期key堆积在内存里，导致redis内存块耗尽了，我们可以设置内存最大使用量，当内存使用量超出时，会施行数据淘汰策略。

两种持久化方式

Redis是基于内存的，如果不想办法将数据保存在硬盘上，一旦Redis重启(退出/故障)，内存的数据将会全部丢失。

  • 我们肯定不想Redis里头的数据由于某些故障全部丢失(导致所有请求都走MySQL)，即便发生了故障也希望可以将Redis原有的数据恢复过来，这就是持久化的作用。

Redis提供了两种不同的持久化方法来讲数据存储到硬盘里边：

  • RDB(基于快照)，将某一时刻的所有数据保存到一个RDB文件中。
  • AOF(append-only-file)，当Redis服务器执行写命令的时候，将执行的写命令保存到AOF文件中。

RDB

RDB持久化可以手动执行，也可以根据服务器配置定期执行。RDB持久化所生成的RDB文件是一个经过压缩的二进制文件，Redis可以通过这个文件还原数据库的数据。

有两个命令可以生成RDB文件：

  • SAVE会阻塞Redis服务器进程，服务器不能接收任何请求，直到RDB文件创建完毕为止。
  • BGSAVE创建出一个子进程，由子进程来负责创建RDB文件，服务器进程可以继续接收请求。

Redis服务器在启动的时候，如果发现有RDB文件，就会自动载入RDB文件(不需要人工干预)，服务器在载入RDB文件期间，会处于阻塞状态，直到载入工作完成。

Redis 还可以通过配置方式定时、定量（变更的 key 达到一定阈值）去创建快照。

AOF

AOF是通过保存Redis服务器所执行的写命令来记录数据库的数据的。

AOF写入步骤

  • 命令追加：把写命令追加写到 AOF 缓冲区；

  • 文件写入：把 AOF 缓冲区的数据写入 AOF 文件，这时候可能是写到 pagecache，操作系统控制什么时候把 pagecache 的数据写到文件；

  • 文件同步：写到 pagecache 其实是在内核态缓冲区，服务器异常可能还是会丢失，所以有把 pagecache 或 AOF 缓冲区的数据强刷到文件的过程。

AOF数据还原

从 AOF 文件中读取并分析每一条命令，然后用伪客户端去执行命令。

AOF重写

为了解决 AOF 文件太大的问题，有了 AOF 重写，重写不会对原有文件进行读取，而是取数据库最新的状态。

AOF 重写可以手动触发，也可以用配置文件的方式，但如果调用函数来执行的话，因为 Redis 是用单线程处理命令的，这样会阻塞线程，导致 Redis 一段时间不可用。

解决这个问题很简单，用一个子进程去执行重写函数就可以了，但是这时候一边重写，一边 Redis 还在接受新的写命令，会造成数据不一致的问题。

为了解决重写后数据不一致的问题，设计了 AOF 重写缓冲区，就把重写期间的写命令写到 AOF 重写缓冲区中，当子进程重写完 AOF 之后，会通知父进程，然后父进程会把重写缓冲区的数据再写到 AOF 重写文件中，这样就保证了重写前后数据一致。

RDB和AOF的选择

RDB和AOF并不互斥，它俩可以同时使用。

  • RDB的优点：载入时恢复数据快、文件体积小，对性能影响少。
  • RDB的缺点：会一定程度上丢失数据(因为系统一旦在定时持久化之前出现宕机现象，此前没有来得及写入磁盘的数据都将丢失。
  • AOF的优点：丢失数据少(默认配置只丢失一秒的数据)。
  • AOF的缺点：恢复数据相对较慢，文件体积大。

如果Redis服务器同时开启了RDB和AOF持久化，服务器会优先使用AOF文件来还原数据(因为AOF更新频率比RDB更新频率要高，还原的数据更完善)

Redis为什么快

  1. 基于内存查询；
  2. 接受命令的是单线程，避免了线程上下文切换；
  3. 接受命令的是核心是 IO 多路复用模型，并且用事件驱动的方式分离了 IO 线程和事件。

多机数据库

为什么要有Redis集群

  1. 数据量大的时候，单机内存放不下；
  2. 单机能够承受的并发量有限；
  3. 单机如果挂了，缓存不可用。

主从架构和哨兵机制解决了单机并发量有限的问题，并且实现了高可用；Redis 集群实现了数据量大单机放不下的问题，以及实现了高并发、高可用。

主从架构

而为了实现高可用，Redis 集群采用主从的架构设计。

  • 主服务器负责接收写请求；
  • 从服务器负责接收读请求；
  • 从服务器的数据由主服务器复制过去。主从服务器的数据是一致的。

复制

复制分为两个阶段：

  • 同步。同步是从服务器向主服务器发送同步请求，让从服务器和主服务器数据一致；
  • 命令传播。主从服务器数据一致之后，主服务器可能因为新的写命令数据又不一致了，这时候主服务器就会用命令传播的方式，让从服务器数据和主服务器保持一致。

同步

同步分为两种：初次同步和断电后同步。

  • 初次同步（2.8 之后也叫做完整重同步）

    • 从服务器发送同步命令给主服务器；
    • 主服务器会调用 BGSAVE 命令，后台生成一个 RDB 文件，并用一个缓冲区记录这段时间的写命令；
    • RDB 文件生成完之后，主服务器发送给从服务器，从服务器根据 RDB 文件去更新数据库状态；
    • RDB 同步完之后，主服务器还会把缓冲区的写命令也发给从服务器，从服务器再更新这段时间的数据库状态。

  • 断电后同步（2.8 之后也叫做部分重同步）

    在 Redis 2.8 之前，断电后同步也需要全量同步，这样浪费性能。所以 2.8 之后是用增量同步的方式。

    部分重同步就是把从服务器断电之后，主服务器的写命令同步给从服务器。

    其实现就是主从都记录同步的偏移量，通过比较偏移量就知道需要同步哪些数据。

命令传播

当完成了同步之后，主从服务器就会进入命令传播阶段。这时主服务器只要将自己的写命令发送给从服务器，而从服务器接收并执行主服务器发送过来的写命令，就可以保证主从服务器一直保持数据一致了！

过期key处理

slave 不会过期 key，只会等待 master 过期 key。如果 master 过期了一个 key，或者通过 LRU 淘汰了一个 key，那么会模拟一条 del 命令发送给 slave。

哨兵机制

主从架构实现了高并发，以及读的高可用，但如果 Master 挂了，写就不可用了，哨兵机制就是实现 Master 挂了之后的高可用。

由多个哨兵实例（特殊的 Redis 实例）组成哨兵系统，然后监视所有主从服务器，当某个主服务器下线之后，会讲下属的一个从服务器升级成主服务器，继续提供写服务，下线的主服务器在后面连上之后当作从服务器。

主备切换的过程

  1. 哨兵初始化

    之后和主从服务器以及其他哨兵都保证连接。

  2. 主观下线

    哨兵会每隔 1 秒和连接的主从服务器以及其他哨兵发送 PING 命令，通过PING命令返回的信息判断实例是否在线。如果主服务器在一定时间都返回无效的回复，那么哨兵就主观认为这台主服务器下线了。

  3. 客观下线

    当哨兵主观认为某个主服务器下线了，那他会询问别的哨兵，如果有足够数量的哨兵都主观认为这个主服务器下线了，那么就是客观下线。

  4. 哨兵选主

    所有监视这个客观下线的主服务器的哨兵都有机会成为领导者，在选主的时候，每个哨兵想其他哨兵发送命令，尝试让别人选自己为领导者，如果得票数超过半数，那这个哨兵就成为领导者。

  5. 领头哨兵进行主备切换

    选择一个下线的 Master 所属的一个 Slave （选一个数据最完备的）作为新的 Master，然后别的 Slave 复制新的 Master，下线的 Master 连上之后，会成为 Slave。

主观下线和客观下线

  • 主观下线

    • Sentinel会以每秒一次的频率向与它创建命令连接的实例(包括主从服务器和其他的Sentinel)发送PING命令，通过PING命令返回的信息判断实例是否在线；
    • 如果一个主服务器在down-after-milliseconds毫秒内连续向Sentinel发送无效回复，那么当前Sentinel就会主观认为该主服务器已经下线了。

  • 客观下线

    • 当Sentinel将一个主服务器判断为主观下线以后，为了确认该主服务器是否真的下线，它会向同样监视该主服务器的Sentinel询问，看它们是否也认为该主服务器是否下线。
    • 如果足够多的Sentinel认为该主服务器是下线的，那么就判定该主服务为客观下线，并对主服务器执行故障转移操作。

哨兵至少需要 3 个实例，来保证自己的健壮性。

哨兵机制实现主备切换的流程

  1. 首先初始化哨兵集群，哨兵本质上就是一个特殊的 Redis 实例；

哨兵主备切换的数据丢失问题

  1. 异步复制导致的数据丢失

    有部分数据还没复制到从服务器，主服务器就宕机了，此时这些部分数据就丢失了。

  2. 脑裂导致的数据丢失

    有时候主服务器脱离了正常网络，跟其他从服务器不能连接。此时哨兵可能就会认为主服务器下线了(然后开启选举，将某个从服务器切换成了主服务器)，但是实际上主服务器还运行着。这个时候，集群里就会有两个服务器(也就是所谓的脑裂)。

    虽然某个从服务器被切换成了主服务器，但是可能客户端还没来得及切换到新的主服务器，客户端还继续写向旧主服务器写数据。旧的服务器重新连接时，会作为从服务器复制新的主服务器(这意味着旧数据丢失)。

集群

架构设计

Redis 集群提供了：

  1. 通过多组 Master/Slave，实现数据分片，水平扩容；
  2. Master/Slave 架构实现高并发；
  3. Master 不可用时，Slave 升级成 Master 的策略实现高可用。

节点间通信

节点之间通过 Gossip 协议通信，一般通信的命令有 4 个：

  • meet。用来进行节点之间的连接；
  • ping。用来检测被 ping 的主节点是否在线；
  • pong。在 ping/meet 之后会回一个 pong，告知在线；
  • fail。当 A 主节点判断 B 主节点下线了之后，A 会向其他主节点发送 fail，其他节点也会把 B 标记为下线。

槽点设计

  • Redis 集群用 hash 的方式来计算数据存放在哪个分片上，设计了 16384 （214）个槽点，然后让集群中的所有 Master 分摊这些槽点，如果所有槽点都有对应的 Master 节点，那集群是可用的。
  • 集群中所有节点都会知道别的节点的槽信息和 IP 端口等信息。

计算键属于哪个槽

  • 用 & 运算代替了 % 运算。
  • CRC16 算法： key*16+key*15+...+key*2+key*1

客户端请求方式

客户端请求一台节点，节点计算 CRC16(key) & 16383 ，key 属于哪个槽，如果是本节点，本节点进行处理；如果不是本节点，返回给客户端 MOVED 错误，里面包含了槽值、槽对应的 IP、端口信息，然后客户端再向对应的 IP 端口发起请求。

重新分片

重新分片一般就是集群中增加了节点，然后需要把一部分槽和对应的数据迁移到新加入的节点。

实现流程：

  1. 目标节点准备倒入槽的键值对（数据）；
  2. 源节点准备迁移槽的键值对；
  3. 每次迁移一部分 count 个键值对，直到该槽的键值对全部迁移完成。

一个一个迁移每一个 slot 槽中的数据，每个 slot 的数据批量迁移。

主备切换

主备切换流程：

  • 每个主节点会定期向其他主节点发送 PING 命令，如果一段时间内没有收到 PONG 命令，就认为这个节点疑似下线（pfail）；
  • 如果超过半数的主节点都认为主节点 X 疑似下线，那么会有一个主节点把 X 标记为已下线（fail），然后这个主节点像别的节点（包括从节点）广播 fail 消息，收到 fail 消息会把 X 标记为已下线；
  • X 下的从节点收到 fail 消息，发现主节点下线了，就会向集群中别的主节点发送消息，要求他们选自己为新的主节点，如果超过半数的主节点都同意了，那就会成为新的主节点；如果没有一个从节点得票超过半数，那么配置纪元自增 1 ，开始新的一轮投票，直到选出新的 Master；
  • 新的 Master 会把槽指派给自己，然后向集群发送 PONG 命令，让别的节点更新信息，然后新的 Master 可以提供服务了。

集群模式和哨兵模式的区别

  1. 哨兵模式监控权交给了哨兵系统，集群模式中是工作节点自己做监控
  2. 哨兵模式发起选举是选举一个leader哨兵节点来处理故障转移，集群模式是在从节点中选举一个新的主节点，来处理故障的转移

缓存雪崩

问题产生原因

缓存雪崩是说，同一时刻缓存都过期了，或者 Redis 服务整个挂了，导致查询压力都直接到了 MySQL，MySQL 处理不了这么多的并发量，导致数据库崩溃。

解决方案

  • 同一时刻大量缓存过期

    在缓存的时候给过期时间加上一个随机值，这样就会大幅度的减少缓存在同一时间过期。

  • Redis 挂了

    事前：用 主从+哨兵，或 Redis 集群的架构方式设计，实现高可用，避免 Redis 服务挂掉；

    事中：万一 Redis 真的挂了，用本地缓存或限流降级的方式，减少数据库并发压力；

    事后：Redis 持久化，重启 Redis 恢复之前缓存的数据。

缓存穿透

问题产生原因

访问一个数据库中不存在的数据，这样永远也不会被放入缓存，如果大量这种请求也会让数据库崩溃。

解决方案

  • 就算数据库中没有这个数据，也把他放入缓存，一般设置个较短的过期时间。

  • 用布隆过滤器来判断查询的数据是否一定不存在。

    • 原理

      把数据库的数据经过多次 hash 之后放入一个只有 0，1 的数组中，每次 hash 都把数组对应的位置设置为 1 ，然后客户端查询的 key 在 Redis 中找不到的话，先在 布隆过滤器中计算一下，也就是进行多次 hash，然后看每次 hash 的结果对应在数组中，是否有 0的，如果有 0，就代表数据库没有这个数据，那就不让他请求数据库了。

      参考博客

    • 特点

      1. hash 出来如果对应位置是 0，那么一定不存在，如果是 1，不能确定是否一定存在；
      2. 影响误判率（本来不存在的结果判断为存在）的有：数组大小和 hash 次数，数组越小，hash 次数越多，很快数组中就都是 1 了，无法判断是否一定不存在；
      3. 不支持删除；
      4. guava 提供了布隆过滤器的快速使用。

缓存与数据库双写一致

一般更新数据库和更新缓存，采用的方式有两种：

  1. 先更新数据库，再删除缓存；
  2. 先删除缓存，再更新数据库。

这两种方式，如果在都成功或都失败的情况下是没有数据不一致的问题的。

为什么用删除缓存而不是更新缓存

  • 高并发的情况下，删除缓存更简单，更新缓存还要考虑并发问题；
  • 可能数据库更新了多次，但这个数据查询半天也用不着，那如果缓存也用更新，耗费了更多不必要的性能，不如删除了之后，在下次查询的时候再放到缓存。

先更新数据库再删除缓存

如果数据库更新成功了，缓存删除失败，那这样就出现数据不一致的情况了。

  • 解决方案

    在低并发的情况下，只需要用先删除缓存再更新数据库就可以了。

先删除缓存再更新数据库

在并发低的时候，如果缓存删除成功了，但数据库更新失败了，这样也不会有数据不一致的问题的。

但在高并发时，如果在删除缓存和更新数据库之间，有别的线程查询这份数据，就会造成缓存和数据库中数据不一致的问题。

时刻 线程1 线程2
t1 删除 key=X 的缓存
t2 查询 key=X，发现缓存中没有，从数据库取旧值，并放入缓存
t3 更新数据库

  • 解决方案

    对于更新操作，把删除缓存和更新数据库，放入一个先进先出的队列中；

    对于查询操作，如果缓存中有那就正常从缓存中取，如果缓存中没有，那把读缓存这个操作放入上面的队列中，这样就不会有从数据库取到旧值的情况了。

其他方案还有用 canal 监听 binlog 同步数据库数据到缓存。
