类加载机制

Java 语言最大的两个特性：跨平台、垃圾收集和内存管理，其中跨平台的实现离不开 .class 文件，class 文件是实现跨平台的基石。下图是一个 Java 文件到堆中的实例对象的过程。本文的类加载机制主要探究的是 Class 文件开始到堆中实例对象的过程。

一个类的生命周期从 class 文件到从内存卸载会经过一下这几个阶段。

加载

加载阶段不要和类加载机制搞混了，加载阶段只是类加载的第一步。

加载的时机

首先我们需要知道什么时候会触发类加载。主要是一下这几种情况：

  1. New 关键字实例化对象，调用类的静态方法、访问类的静态字段(被 final 修饰的不算，直接存放在 class 文件的常量池中了)；
  2. 通过反射调用一个类；
  3. 当初始化一个类时，如果其父类还未初始化，那么也会去加载这个父类；
  4. 虚拟机启动时，加载制定的 Main() 的那个类。

加载的主要任务

  1. 根据类的全限定名获取类的二进制流，这里的二进制流主要就是 class 文件，当然也包括 jar、war 等别的形式的二进制流；
  2. 把这个二进制流转换成方法区中的数据；
  3. 在方法区创建一个代表这个类的 java.lang.Class 对象，作为访问这个类的入口。

连接

连接阶段分为三部分：验证、准备、解析。

验证阶段就是对 class 文件进行各种校验，没什么好说的，都是 JVM 控制的。

解析阶段发生时间并不一定，可能会在初始化阶段之后，主要的工作就是对 class 文件进行各种解析，也不需要关注。

准备阶段是会为类的静态变量(static 修饰)分配内存，并赋上默认值，这里的默认值需要注意，被 final 修饰的静态变量默认值就是其值。类的实例变量是在初始化阶段分配内存的，并且初始化阶段还会执行静态变量的赋值。

连接阶段都是虚拟机控制的，开发者只需要关注加载(什么时候去加载)、和初始化阶段。

初始化

初始化阶段其实就是执行类构造器 < client > 方法的过程。< client > 方法会收集类中所有的类变量的赋值动作(例如，static int a = 3)以及所有的静态代码块(static{})，并且是按照在文件中的顺序来的。

注意类加载机制中的初始化和实例化的区别：

  • 在类加载机制的准备阶段就会为类的静态变量分配内存，是在方法区上分配的，而初始化阶段就和名字一样，就是为这些类变量设置值，设置值就是上面说的执行 < client > 方法的过程；
  • 实例化之前会判断是否完成了类加载(也就是初始化)，没有的话才需要去进行类加载，一个类初始化(类加载)执行一次，实例化可以执行多次(创建多个类)。实例化是执行实例构造方法 < init > 的过程，就是构造方法。

这里强调一下类变量、常量、实例变量这几个的区别，我们还是以上面的例子来说明：

类变量：类中 static 修饰的变量，是在类加载准备阶段就分配了内存在方法区，在初始化阶段进行赋值；

常量：类中 static final 修饰的变量，在类加载准备阶段就分配并赋值了，在方法区上；

实例变量：类中没有 static 修饰的变量，在实例化之后(执行了构造方法)就会在堆上进行分配，并赋值。

类加载器

类加载机制中还有一个重要的东西就是—类加载器。类加载器是实现加载的工具。类加载器分为3类：

  1. 启动类加载器；
  2. 扩展类加载器；
  3. 应用类加载器。

此外，还需要知道的是类加载器的双亲委派模型，可以看我以前的类加载器的博客。
