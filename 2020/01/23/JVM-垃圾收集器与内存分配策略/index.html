JVM-垃圾收集器与内存分配策略
发表于 2020-01-23| 更新于 2020-02-01|JVM垃圾收集器

判断对象是否需要回收

引用计数算法

引用计数算法就是，对象被引用就计数 +1，然后引用失效就计数 -1。这种方式简单，但是存在循环引用情况。循环引用就是 A 引用了 B，B 引用了 A，这样这两个对象永远都不会被回收。

可达性分析

大多数 JVM 都是采用了可达性分析的方式来判断对象是否需要回收。可达性分析是从一系列 GCRoot 开始沿着引用链寻找，如果某个对象没有和引用链上的任何对象关联，这个对象就是不可达的，就是需要被回收的。

GCRoot

可达性分析从 GCRoot 开始，那哪些对象可以作为 GCRoot 呢？主要分为两类：

  • 全局性的引用。方法区中的常量、静态属性引用的对象(就是定义的常量以及类变量)；
  • 执行上下文。栈中的局部变量表所引用的对象(就是运行的方法的局部变量)。

引用的类型

无论是引用计数、可达性分析算法都依赖“引用”的实现。在 Java 中引用不是简单的有引用、没有引用两种状态，Java 中一共有4种引用方式。

强引用

强引用的对象不会被 GC 回收，如果内存不够了，抛出 OOM 异常也不回收。

强引用就是我们最通常使用的 new 一个对象。

软引用

软引用的对象一般也不会被 GC 回收，但在内存不够时，这些软引用的对象才会被回收。

java 中提供了 SoftReference 来实现软引用。

弱引用

弱引用的对象在发生 GC 时，就会被回收(不会对引用对象的存活产生影响，该回收就回收)。

Java 中提供了 WeakReference 来实现弱引用。

虚引用

虚引用的对象在发生 GC 时，就会被回收(不会对引用对象的存活产生影响，该回收就回收)，甚至无法通过虚引用来获取引用对象。

Java 中提供了 PhantomReference 来实现虚引用。

如何回收

主流的虚拟机都是用可达性分析的方式来判断对象是否需要回收，从 GCRoot 开始沿着引用链找，没有和引用链关联的对象就会被回收，原理是这样的，但在实现上我们还需要扣扣细节。

枚举根节点—OopMap

程序运行时执行上下文(栈中的局部变量表)、全局性引用(方法区静态属性和常量)数据量可能很大，那当要 GC 的时候，我们遍历整个虚拟机来获取这些 GCRoot 就会花费大量的时间。更重要的是，在获取这些 GCRoot 的时候，是需要 Stop The World 的，而如果我们获取根节点的时间很长，那会极大的影响 GC 效率。

为了解决枚举 GCRoot 耗时长的问题， Hotspot 是用 OopMap 来解决的，OopMap 中存放的是一个对象内什么偏移量上是什么类型的数据。通过这种方式我们就能快速定位这个对象中的 GCRoot。

现在主流的虚拟机基本都采用准确式的 GC，准确式 GC 就是明确了某块内存对应的数据类型是什么，这样在 GC 的时候就可以知道这块空间是不是引用类型。Hotspot 的准确式 GC 就是基于 OopMap 实现。

GC发生的位置—安全点和安全区域

GC 并不是在任何位置都能发生，只会在安全点/安全区域才会发生 GC，这是因为如果每行指令都生成 OopMap，那占用空间太大，所以只需要在特定的地方去生成 OopMap，进行 GC。安全点一般是在一下这几个地方：

  • 循环的末尾；
  • 方法返回前；
  • 抛出异常的位置。

这样我们要进行 GC 时，就是在安全点暂停所有线程，然后执行 GC 线程。但如果让所有线程在安全点暂停也是一个问题，有两种方式：

  1. 主动中断，现在都用这个。在安全点设置标志位，所有线程到安全点的时候去查询标志位，如果标志位需要中断，线程就中断在安全点了。
  2. 抢先中断。先让所有线程都中断，如果有线程不在安全点，让他跑到安全点在中断。

如果线程原本就是休眠或阻塞的状态，那肯定也不能唤醒他，然后让他跑到安全点。为了针对这种情况，有了安全区域的概念，线程在安全区域内任何地方都可以发起 GC，其实就是扩大版的安全点。

最后的机会—finalize

如果一个对象从 GCRoot 的引用链开始都没有关联，那这个对象是需要被我们回收的。但虚拟机也并不是立刻就会回收，对于没有和 GCRoot 关联的对象会先进行第一次标记并进行筛选。

  • 如果对象的 finalize() 没有执行过，那么在第一次标记之后会把这个对象放入一个队列中，虚拟机会启动一个线程执行这个队列中对象的 finalize() 方法；

    • 如果在执行 finalize() 过程中，重新和引用链上的对象关联上，那这个对象就不会被回收；
    • 如果执行完 finalize() 没有和引用链关联，那这个对象会被进行第二次标记；

  • 如果对象已经执行过 finalize() 或执行之后还是没和引用链关联，那这个对象会被进行第二次标记，然后第二次标记的对象就需要被回收。

垃圾回收算法

标记-清除

先标记后将标记的对象回收。

缺陷：产生内存碎片。

复制算法

内存分为两个区域，80%的 Eden 区和两块10%的 survivor 区。对象先分配在 Eden 和一块 survivor 上，然后 GC 的时候把还存活的复制到另一个 survivor 上，原来的 survivor 和 Eden 全部清除。

缺陷：空间利用率低了；在复制的时候可能会分配空间不够，这时候需要分配担保。

标记-整理

跟标记-清除类似，只不过是标记-清除之后，还存活的对象会移到同一边。解决了内存碎片的问题。

在 Java 虚拟机中都采用分代垃圾收集的方式。将堆内存分为新生代和老年代，新生代因为对象大多都是朝生夕灭，采用复制算法，新生代分为 Eden 和两个 survivor 区，而老年代无法使用分配担保，采用标记-清除或标记-整理算法。

分配担保是说复制算法的时候复制的时候，如果 survivor 的空间不足，那么就会用老年代的空间来做担保，会把分配不下的放在老年代中，而如果老年代也放不下那么就会触发 Full GC。

垃圾收集器—垃圾回收的具体实现

上面的部分都是垃圾收集的基础算法、原理，垃圾收集器才是真正工作的回收垃圾的，垃圾收集器我们重点关注两个就行了：CMS 和 G1。

CMS

传统的垃圾收集器由于标记回收对象的时候，不可避免的需要 Stop The World，所以会有较大的停顿时间，这是使用者不能忍受的。

CMS 是以更少停顿时间为目的而设计的，其最大的特点是标记和清除一定程度上可以和用户线程一起工作，且 CMS 是以标记-清除算法为基础的。CMS 工作流程如下：

  • 初始标记。需要 STW，但只标记 GCRoot 直接关联的对象，时间很短；
  • 并发标记。可以和用户线程并发执行，这时候就会根据引用链去标记，时间长；
  • 重新标记。需要 STW，标记的是并发标记期间变更的引用；
  • 并发清除。可以和用户线程并发执行。

CMS 是一款优秀的垃圾收集器，使用广泛，但还是存在一下3个缺陷：

  1. 虽然不会让用户线程停顿，但和用户线程并发工作，占用 CPU 资源，吞吐量不高；
  2. 在并发清除阶段，清除线程和用户线程并发执行，不可避免的用户线程会产生别的垃圾(也称为浮动垃圾)，这些垃圾只能等到下次 GC 在回收，并且由于浮动垃圾的存在，需要预留足够的内存空间给用户线程使用，也就是本来100%的老年代，那可能用到 80% 就需要触发 GC 了，剩下的空间预留给并发清除的用户线程；
  3. CMS 基于标记-清除算法实现，所以存在内存碎片的问题。解决方案：CMS 提供了参数 UseCMSCompactAtFullCollection，在发生 FullGC 的时候会去合并内存。

G1

G1 和 CMS 一样都是专注于减少停顿时间的垃圾收集器，但 G1 和之前的垃圾收集器在内存分布等上有很大的不同，其特点如下：

  • G1 将堆内存“化整为零”，分为一个个小的内存区域，但依然保留了分代收集的策略，这么做也是更好的利用对象；

  • G1 从整体上看使用的 标记-整理 算法，从局部看两个内存区域之间是 复制 算法，所以 G1 不会产生内存碎片；

  • G1 充分利用并发和多核 CPU 的特性，减少 STW 的时间；

  • G1 提供了可预测停顿时间的功能。G1 可预测停顿时间的实现最根本的原因就是“机智”的将堆内存分为一块块的区域，这样我们回收垃圾可以只回收部分区域，并且通常回收的都是利用率最高的那些区域，而不是每次都回收整个堆空间，这样就做到了停顿时间的可控。

G1 和 CMS 一样都采用并发标记和清除的方式让回收线程和用户线程并发执行。而 G1 回收的时候会对每个区域的垃圾比率做计算排序，优先收集垃圾多的区域。

对象分配的几条原则

对象分配其实就是给对象在堆上分配内存，一般都是分配在 Eden 区，如果用了本地线程缓存，那么对象优先分配在本地线程缓存上，随着优化技术的发展，也有栈上分配对象的情况。

  1. 堆上 Eden 分配；
  2. 堆上本地线程缓存分配；
  3. 栈上分配(对象没有方法逃逸，就可以用栈上分配)。

本地线程缓存是这样的：对象在堆上分配内存空间是线程共享的，所以可能会有多个线程同时分配同一块内存的问题，所以一般分配在堆上是需要线程同步的，这样效率低。如果用了本地线程缓存，那么就会在堆空间中，开辟一段给这个线程的空间，这样这个线程分配对象就直接分配在这个空间上，不用加锁。

然后下面说几条对象分配的原则。

  • 对象优先分配在 Eden 区

  • 大对象直接进入老年代

  • 长期存活的对象进入老年代

  • 分配担保

最后再来说说年轻代和老年代发生 GC 的时机和名称。

其实发生 GC 的时机都是分配对象的时候发现空间不足了，而如果是年轻代分配空间不足，那么触发年轻代的垃圾回收 Minor GC，速度较快；如果 GC 发生在老年代，成为 Full GC/Major GC，时间较长，一般也会伴随 Minor GC。
