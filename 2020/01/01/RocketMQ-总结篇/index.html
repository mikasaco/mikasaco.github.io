RocketMQ-总结篇
发表于 2020-01-01| 更新于 2020-03-04|MQRocketMQ

消息中间件的优点

消息中间件最主要的优点是以下三点：

  • 解偶

    几个系统之间协作，如果没有用消息中间件，用接口调用的方式。例如订单系统下单接口，下单完成之后需要调用库存系统的接口去减库存，调用支付系统完成余额扣款，但如果此时又上线了一个物流系统，下单之后之间生成物流单子，那是不是就得去改订单系统下单接口呢？

    如果用了消息中间件，那我们可以把下单的消息投递给消息队列，然后订单系统就可以返回了，后面的物流系统、支付系统、库存系统都订阅下单的 topic，然后自己去消费就可以了。

  • 异步

    再接着上面的例子，如果没有消息中间件，那么用户下单之后订单系统需要进行一些操作，并在这个接口里还要去调用支付、库存等系统的接口，总耗时长。

    如果用消息中间件，那用户下单之后，只需要往消息队列中投递消息就可以返回了，支付、库存等系统的处理从消息队列中取消息，异步处理即可，耗时短。

  • 削峰

    消息中间件还可以用来做类似缓存的设计，当系统流量太大时（峰值）可以把数据放到消息中间件中，然后后端应用慢慢（限流）处理消息队列中的消息，不至于流量太大把系统压垮。

架构和术语

RocketMQ 中架构上有 4 种角色：producer , Name Server , broker , consumer 。具体的各个角色的组件在后面再详细分析。

Broker集群

Broker 集群存储消息，以及接收消费者、提供者的请求，Broker 集群包括多组 Master/Slave ，Master 可读可写， Slave 只可读，Master 会将写入的数据同步给 Slave。每个 Broker 启动之后会把自己的信息上传给每个 Name Server ，并和所有的 NameServer 保持长连接，并定期上传 topic、queue 等信息。

Producer集群

消息的生产者，通过 NameServer 集群获得 Topic 的路由信息，包括 Topic 下面有哪些 Queue ，这些 Queue 分布在哪些 Broker 上等。 Producer 只会将消息发送到 Master 节点上，因此只需要与 Master 节点建立连接。

Consumer集群

消息的消费者，通过 NameServer 集群获得 Topic 的路由信息，连接到对应的 Broker 上消费消息。注意，由于 Master 和 Slave 都可以读取消息，因此 Consumer 会与 Master 和 Slave 都建立连接。

Name Server集群

Name Server 为消息生产/消费提供路由信息。Name Server 集群每个 NS 彼此之间是不同通信的，且每个 NS 都会和所有的 Broker 保持长连接，存有 Broker、topic、queue 信息。

生产者/消费者

消费者

消息获取方式（push和pull）

RocketMQ 消费者有两种消费方式： pull 和 push ，对应 DefaultMQPullConsumer 和DefaultMQPushConsumer ，我们一般用 DefaultMQPushConsumer ，前者需要自己维护偏移量。

消息获取方式 优点 缺点
Push—服务端在收到消息后推给订阅的客户端 消息可以更加即时的给到订阅的客户端 服务端不知道客户端的消息处理能力，可能推的消息过多，把订阅消息的客户端压垮
Pull—客户端轮询主动从服务端拉取订阅的消息 客户端可以自己把控消息拉取的速度 客户端不知道什么时候去拉取消息，存在消息传达不及时的问题

RocketMQ 的 push 方式兼顾 push 和 pull 的优势。他是这样的：客户端从轮询拉取服务端订阅的消息，但服务端收到客户端的拉消息请求后，如果有新消息就理解返回给客户端，如果没有新消息，服务端会循环查询 15s （相当于 hold 住这个请求），如果 15s 内有新消息，就立刻返回给客户端，如果超过 15s 没有返回空结果。

这种方式也叫做长轮询，这种方式的缺点就是这 15s 是浪费了服务端的性能的，适合订阅服务连接数可控的情况。

消息模式（广播模式和集群模式）

对于同一个 consumer group 的消费者来说，有两种消息模式：

  1. 广播模式。一个 topic 中的每条消息会被 consumer group 中的每个 consumer 都消费一次；
  2. 集群模式。一个 topic 中的每条消息只会被 consumer group 中的一个 consumer 消费一次（ consumer group 中所有的 consumer 合起来消费一个 topic ，这时候是负载均衡消费的）。

生产者

生产者比较简单，一般使用的 API 就是 DefaultMQProducer ，其消息发送可以有同步和异步两种方式。

消息发送到 topic ，但具体在 topic 的那个 queue 是未知， RocketMQ 会根据负载均衡策略把消息存放在 一个 topic 中的 queue 中。我们可以使用 MessageQueueSelector 来自主控制发送给哪个 queue。

NameServer

NameServer 的作用就是给 producer 和 consumer 提供 Broker、topic、queue 的路由信息，而需要注意的是 NameServer 集群中各个 NameServer 互相不通信，每个 NameServer 都会保存所有的 Broker 的信息，包括 topic、queue、IP。

为什么不选用Zookeeper

RocketMQ 的架构设计就不需要用 zk ，NameServer 集群中每个节点都会保存所有的 Broker 信息，不需要 Master 选主这些 zk 的功能，只要提供轻量的元数据服务就可以。

上面这句话其实也可以理解为 NS 采用的是 AP 的策略，不同于 zk 采用的 CP 策略。

Kafka 需要用 zk 来对 Broker 选主，选出一个 Master Broker 对 Broker 进行管理，管理包括 Patition选主、集群状态等。

NameServer的CAP理论

CAP 理论就是分布式系统中数据读写一致性、可用性、分区容错性三者不能同时满足。

C 一致性：同一时间从集群中获取的数据都是一致的。

A 可用性：部分节点故障，整个系统还是能够提供服务。

P 分区容错性：分布式系统可以容忍节点之间网络故障。

对于分布式系统，分区容错性是最基本的，因为既然是分布式系统那肯定有多个节点，而网络异常是不可知的，所以必须要保证系统可以容忍节点之间的网络问题。所以一般都是满足 CP 或 AP 。

其实 CP 和 AP 也很好理解。当分布式系统中部分节点发生故障时，

  • 如果允许数据写入，那此时就实现了可用性，但故障的节点和别的节点就会出现数据不一致的问题，这就是满足了 AP ，放弃了 C；
  • 如果不允许数据写入，那此时就实现了一致性，但系统不可用了，这就是满足了 CP ，放弃了 A。

对于 NameServer 就是典型的 AP 。NameServer 放弃了一致性，也就是某个时刻各个节点的数据可能不一致，但 NameServer 保证最终获取到的数据是一致的。那 NameServer 又是怎么保证最终数据一致的呢？

NameServer如何保证数据的最终一致

NameServer 主要从3个方面去保证数据的最终一致性： Broker 注册、 Broker 销毁、consumer/producer 的路由发现。

Broker 注册

Broker 会在启动的时候获取所有的 NameServer 列表，然后和每个 NameServer 建立长连接，并每隔 30s 上传心跳包，里面会有 Broker 的信息。

Broker 销毁

正常情况下， Broker 销毁会断开和 NameServer 的长连接，那 NameServer 就知道这个 Broker 销毁了，就可以剔除这个 Broker（每个 NameServer 都会这么去剔除）。

而当 Broker 异常销毁时，NameServer 还维持着长连接，但 NameServer 会有一个后台的定时任务，每隔一段时间检查 Broker 列表的情况，如果 Broker 的最新的心跳包超过了 120s ，那就当这个 Broker 也是销毁了，就会从 Broker 列表中剔除。

路由发现

无论是 consumer/producer 都会有一个定时任务去从 NameServer 拉 Broker 的数据，然后在缓存在本地，然后根据缓存在本地的 Broker 信息去消费/生产。但这中间存在一定的延时（Broker 变更在下次拉取之前存在数据不一致），这时候就用客户端的重试机制来保证。

客户端的重试机制很重要，虽然 NameServer 集群不保证同一时刻不是所有的节点的数据都一致的，但可以通过重试机制来获取到可用的 Broker。

Broker

Broker 是 RocketMQ 的核心，这部分我们需要关注：消息存储方案、Broker 的 HA 机制。

消息存储方案

作为消息中间件，消息要持久化在磁盘上，那怎么保证读写的性能的？ RocketMQ 以顺序写、随机读的消息存储结构，并结合“零拷贝”技术，提高了消息读写的性能。

零拷贝

RocketMQ 提升消息读写性能的“法宝”之一就是使用了零拷贝技术，具体使用的是内存映射的方式去实现零拷贝。

消息存储结构

RocketMQ 消息存储结构主要是两部分：实际存储消息的 Commit Log 和 存储 topic 队列的 Consume Queue，在 Consume Queue 中存储的是 Commit Log 的消息偏移量，所有的消息（any topic , any queue）都会存在 Commit Log 中，而在 Consume Queue 消息是有序的。

顺序写

RocketMQ 的顺序写主要说的是 producer 发送的消息写到 Commit Log 是用追加的方式。

生产者面向的是 Commit Log ，当需要增加一条消息到时候，RocketMQ 的处理流程如下：

  1. 因为 Commit Log 已经通过内存映射 mmap 的方式在用户态建立了虚拟内存，所以我们追加一条消息，都不用切换到内核态，用户态直接操作虚拟内存的虚拟地址就可以了；

  2. 虚拟地址通过内存管理 MMU 映射到内存中真实的物理内存地址，也就是 page cache ，

    • 如果对应的 page 存在，那直接往这个 page 上写数据就可以了，这时如果是异步刷盘，已经可以返回写入成功给生产者了；
    • 如果对应的 page 不存在，那就需要再把 Commit Log 对应的数据页加载进 page cache，然后再往 page 上写数据。这时候性能就差了，因为需要磁盘 IO，所以需要顺序写，因为可以最大化利用 page cache 的预读特性。

  3. 写到了 page cache 对于异步刷盘，就可以返回写入成功给生产者了；而对于同步刷盘还需要把 page cache 的数据刷到磁盘文件才返回。

如果把 Commit Log 整个文件都加载进了内存，那写消息的时候，无论是顺序写还是随机写都是内存操作，应该性能也好，为什么 RMQ 还要实现顺序写？

我的理解是可能针对内存放不下整个 Commit Log 文件（1G）的情况吧。

异步刷盘和同步刷盘
刷盘方式 优点 缺点
异步刷盘 性能上更快，大多数场景的选择 如果服务器断电在 page cache 的数据就会丢失
同步刷盘 不会丢失消息 性能差，适合支付这种完全不能丢失消息的场景

如果 Broker 应用宕机不会丢失消息，在 page cache 的数据会有 os 刷到磁盘的。

随机读

RMQ 的随机读，主要说的是在 Commit Log 上是随机读的。

当消费者读消息时，面向的是 Consume Queue，在队列上消费，也是顺序读的，而且 Consume Queue 数据量不大，完全可以整个读入内存，所以 Consume Queue 上的读也是很快的。

关键在于从 Consume Queue 取到 Commit Log 中的偏移量后是随机读的。但还是可以通过 page cache 的机制优化读。

我感觉这里对于 1G 的 Commit Log 通过内存映射，其整个文件都可以在内存中，所以这时候的随机读还是基于内存的操作。

Broker的HA机制

为了实现高可用，Broker 采用的是主从的架构。Master 可读可写，Salve 只可读，RMQ 不支持 Salve 在 Master 挂了之后自动升级为 Master 的设计。

  • 对于 consumer ，如果 Master 挂了，那读消息可以从 Salve 读，实现了 HA；

  • 对于 producer ，需要在创建 topic 的时候在多个 broker 组上创建，也就是写到了多个 Master 上，这样一个 Master 挂了不能写，通过重试机制还可以写到别的 Master 上，实现了 HA。这里一个 topic 多个 queue 可以分布在多台 broker 上，也解决了一个数据量很大的 topic 不能在单机存储的问题。

上面的总结来说，当 Master 挂了之后可以在这个 Broker 组上继续消费（消费的是 Salve），但无法往这个 Broker 组写消息。

RocketMQ4.5 版本之后还提供了一种多副本机制，DLedger，支持 Master 挂了之后，Salve 提升为 Master 继续提供读写服务。这个机制的 Salve 至少 2 个，因为用的选主算法是 Raft 协议。

主从同步方式

对于 Master 什么时候把 producer 生产的消息同步到 Salve 有两种方式：同步复制和异步复制。

主从同步方式 优点 缺点
同步复制 Master 把消息同步到 Salve 才返回写入成功 可靠性高 延时高，吞吐量低
异步复制 Master 写入成功就返回 延时低，吞吐量高 如果 Master 挂了造成 Master 和 Salve 数据不一致

一般设置主从同步方式为同步复制，刷盘方式为异步刷盘。

顺序消息

RMQ 中顺序消息两种：全局顺序消息和部分顺序消息。

全局顺序消息那我们设置 topic 的 queue 是 1 ，这样这个 topic 下的消息都在一个队列，那肯定是有序的。

部分顺序消息那发送消息的时候忘同一个 queue 中发送就可以了。

重复消息

RMQ 是至少一次的消息投递方式，也就是确保不会丢消息，但可能有重复消息。

那怎么处理重复消息呢？老规矩，业务幂等。
