AOP的使用

什么是 AOP？

如果用面向对象 OOP 的编程思想，我们给一些类增加一个相同的功能（这个功能还是从属于某一个类的，只是每个类都有相同的功能），需要在每个类中都增加这样的功能。那如果这个功能要修改一下呢？或者再增加一个公共的功能？

这时候 OOP 的思想就会带来问题了，AOP 面向切面的编程思想就可以很好的解决这种问题。AOP 可以给这些类一个统一的切面，我们需要增加的功能都可以围绕这个切面进行编程，而这个切面是适合一系列类的。

OOP 是纵向结构，AOP 是横向结构

AOP和AspectJ

AOP 只是一种编程思想，在 Spring 中 AOP 的具体实现就是 AspectJ。

JDK动态代理

创建代理类

JDK 动态代理是通过反射 API ProxyGenerator.generateProxyClass() 生成代理类的字节码文件并加载，然后实例化代理类，例如： $proxy13。我们看下生成的代理类到底是什么，以及是怎么去代理我们的原来的方法的。

上面 $proxy 类中可以看到：

  1. 其构造方法中会传入一个 InvocationHandler 类，熟悉 JDK 动态代理肯定知道这个类就是实现动态代理的关键；
  2. 代理的 login() 方法实际是调用了 InvocationHandler.invoke() 方法。

在 Spring AOP 中这个 InvocationHandler 就是 JdkDynamicAopProxy，我们在看一下最后被代理的类。

使用代理类

当我们执行 userService.login() 时，其实际执行的类调用关系其实是如下面的这样的：

所以对于 JDK 动态代理其执行过程是

  1. 调用代理类 $Proxy 的代理方法 login()；
  2. $Proxy.login() 中会调用 JdkDynamicAopProxy.invoke() 方法；
  3. JdkDynamicAopProxy.invoke() 最终会把各种增强（包括 before，after，around 等）放在一个责任链上，然后在 ReflectiveMethodInvocation.proceed() 方法中递归调用这个责任链，如果某个增强有配置到这个方法就会执行这个增强的功能（例如本文例子中的 MethodBeforeAdviceInterceptor.invoke()）；
  4. 最后会执行切面的方法，并且通过反射调用被代理类的方法（本文例子中的 userService.login()）。

切面方法并不会放在责任链上。

CGLib动态代理

CGLib 和 JDK 的区别就是从上面 createAopProxy() 确定代理方式开始的，CGLib 的源码没有仔细深究，最主要的先知道 CGLib 是没有接口的时候创建动态代理的方式。

CGLib是对指定的类生成一个子类，覆盖其中的所有方法来实现增强的，也因此所以该类代理的方法不能声明称final的。

JDK和CGLib的比较

JDK 创建代理类快，但运行代理类使用反射的，速度慢；

CGLib 创建代理类慢，但运行代理类快。
