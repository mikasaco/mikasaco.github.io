MQ-kafka和RocketMQ对比
发表于 2020-01-10| 更新于 2020-02-13|MQkafka和RocketMQ对比

前言

RocketMQ 在设计上借鉴了 kafka ，但支持更多的 topic ， 两者的差别主要有以下这几个方面：

  1. 编程语言。kafka 是 Scala 语言实现的，RMQ 是 Java 语言实现的；
  2. 性能。kafka 单机性能百万级 TPS，RMQ 小一个数量级，但 RMQ 可以支持的 topic 数比 kafka 多一个数量级；
  3. NS 和 ZK；
  4. Broker 设计（HA、主从、零拷贝）；
  5. consumer 消费方式。

下面主要对后面三点进行具体的介绍对比。

NS 和 ZK

kafka 使用 zk 来管理 Broker、topic 等信息，在 kafka 架构的 zk 集群中，broker 注册到 zk 的话是和 zk 中的 Master 交互的，然后 Master 在同步给其他 Salve。实现了 CAP 理论的 CP ，也就是保证了 zk 集群中数据的一致性。

而 RMQ 中 NS 之间没有主从关系，甚至 NS 之间互不通信，所以 NS 集群之间是会存在数据不一致的情况的，他实现的是 CAP 理论的 AP，也就是保证了可用性。

kafka 为什么需要用 zk 呢？一个当然是提供 Broker 、topic 路由信息等，另外也提供 Broker 选举功能。

Broker设计

Broker 设计的区别是 Kafka 和 RMQ 区别最大的体现了，由于 Broker 设计的不同也带来了两者高可用 HA 实现的不同，主从设计的不同，topic 数量不同等的原因。我们看下两者 Broker 的区别，也可以说是数据存储的不同。

单机角度

Kafka 消息数据存储在每个分区 patition 中，每个 topic 对应着几个 patition；

RMQ 把这台 Broker 上的消息数据都存储在 Commit Log 上，一个 topic 对应着几个逻辑的 consume queue。

此外对于零拷贝的实现也不相同，RMQ 基于内存映射 mmap，kafka 基于 sendfile。

集群角度

主从设计&HA实现区别

kafka 的主从关系说的是 patition；RMQ 的主从关系说的是 Borker。

kafka 中 Borker 集群虽然有一台 Master Borker ，但他是管理 Borker 集群的，不是做数据备份的，patition 是有主从的，Salve patition 是 Master patition 的备份，这种方式来实现 HA。当 Master patition 所在的 Borker 故障时，Salve patition 会晋升为 Master patition。 （有点像 ES 的分区设计）

RMQ 中 Borker 集群中可以有多个 Borker 组，每个 Borker 就是主从关系，Salve Borker 是 Master Borker 的备份，这种方式实现 HA。当 Master Borker 故障时，Salve Borker 不会晋升为 Master Borker ，但消息消费可以从 Salve Broker 上读，消息生产的 HA 就是把一个 topic 放在两个 Broker 组上实现。
