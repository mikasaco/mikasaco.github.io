IO-bufferIO,DirectIO,内存映射
发表于 2019-12-07| 更新于 2020-02-13|IO零拷贝

BufferIO

BufferIO 也就是操作系统标准的 IO 方式，最大的特点是会使用操作系统的 page cache 页缓存。

在从应用系统写数据到磁盘的时候，是会先写到 page cache，这样对于应用程序来说，写操作就完成了。然后操作系统会在合适的时候把 page cache 中的数据再刷到磁盘文件上。应用程序也可以调用操作系统 flush() 函数，page cache 数据刷到磁盘文件之后再返回写操作完成。

在应用程序从磁盘读的时候，是会先从 page cache 中读，如果命中了，那就不需要再到磁盘去读了，减少了到磁盘的 IO 操作，如果没命中，会把数据所在的那一页都读入 page cache。

优点：读数据命中缓存可以减少 IO 次数，写数据可以快速返回，性能高。

缺点：需要在用户缓存和操作系统 page cache 中多次拷贝，在有些场景浪费性能。

对应的在 Java API 中 HeapByteBuffer、DirectByteBuffer 都是 BufferIO。HeapByteBuffer 是在缓冲 buffer 在堆内，DirectByteBuffer 是缓冲 buffer 在堆外。

DirectIO

对于 Direct IO，读写不会用到 page cache，写就直接写到磁盘，读也是直接从磁盘读。

Direct IO 特点：少了 page cache 的拷贝，但读写没用到 page cache。

在 Java API 中 DirectByteBuffer 并不是 DirectIO，这个用的很少。

零拷贝

零拷贝和上面两个 IO 其实不是同一个概念，零拷贝更多的是说减少用户缓冲区和内核缓冲区的拷贝，对应的 API 是 Channel.transferTo() 和 MappedByteBuffer，而上面两个 IO 说的是读写是否经过 page cache ，对应的 API 是除了 MappedByteBuffer 的 ByteBuffer。

对于传统 IO（BufferIO）需要在用户态和内核态来回复制缓冲区的数据。

Linux 实现零拷贝有两种：sendfile() 函数和 mmap 内存映射。

零拷贝不是说不用拷贝了，而是不用在用户态和内核态直接拷贝数据。

sendfile()

sendfile() 函数对应的 Java API 就是 Channel.transferTo() 函数。sendfile() 函数可以讲内核态缓冲区中的数据指定发送给某个地方，这样就不用再拷贝到用户态了。

优点：磁盘文件数据拷贝到内核态的缓冲区，直接传输到内核态别的地方（例如网卡上发送数据），避免了再拷贝到用户缓冲区，和从用户缓冲区拷贝到内核缓冲，实现了“零拷贝”。

缺点：适合将磁盘文件发送出去（磁盘文件复制到网卡）的场景，用户不能操作数据。

内存映射

针对上面 sendfile() 不能操作数据的问题，又有了内存映射的零拷贝方案，对应的 Linux 函数是 mmap() ，对应 Java API 是 MappedByteBuffer。

内存映射的原理是：把磁盘文件映射到虚拟内存，虚拟内存不是真正存储磁盘文件数据的，而是需要将虚拟内存中的虚拟地址通过 MMU（内存管理）再映射成真实的物理地址，再把文件加载进物理内存（也就是 page cache）。接下来用户程序只需要在用户态操作虚拟内存，通过 MMU 就可以映射到 page cache，完成对文件的操作，这样就减少了用户态和内核态的切换，也不用用户态和内核态数据的拷贝了。

CPU芯片上有个内存管理单元就是 MMU，可以把虚拟地址转换成物理地址。

虚拟内存是操作系统为每个进程都会分配的。

用户态和内核态

说了很多的用户态和内核态，这里简单介绍一下。

内核态：cpu可以访问内存的所有数据，包括外围设备，例如硬盘，网卡等。

用户态：只能受限的访问内存，且不允许访问外围设备。

像文件读取等我们是需要磁盘设备配合的，在用户态不能实现，所以需要切换内核态和用户态，而这种切换是需要消耗性能的。
