MySQL-Join查询原理和优化
发表于 2019-11-02| 更新于 2019-12-04|MySQL查询原理和优化join查询

一图概括

前言

经常看到会说不要使用连表查询，连表查询性能很差之类的话，那为什么连表(join)查询性能差呢？我们到底能不能用Join查询呢？Join查询有什么优化的方法吗？本文带着这几个问题，分析一下连表查询的原理和优化。

以下面两张表user表和role表为例进行分析,在role和user表中建了station_id的索引。

MySQL join查询原理

对于MySQL的连表查询都是走的Nested-Loop Join算法，也就是循环遍历匹配的方式，但根据是否用了索引，又可以分为Index Nested-Loop Join和Block Nested-Loop Join。

走索引的join查询(Index Nested-Loop Join)

我们以这条SQL为例，驱动表是user，被驱动表是role，role表中对station_id加了索引，查看一下这句SQL的查询计划，结果如下。

可以看到对于role表走的是k_station_id的索引。那么这句查询SQL在MySQL到底都做了什么呢？

这句SQL user表做驱动表，role表做了被驱动表，并且对于user表，虽然有索引station_id，但并没有用，走的是全表扫描，对于驱动表需要全表扫描，role表走的是station_id的索引，SQL执行的流程：

  1. 从user表中取一条数据R；
  2. 从R中找到字段station_id到role表中查询；
  3. 取出表role中满足条件的行，跟 R 组成一行，作为结果集的一部分；
  4. 重复执行步骤 1 到 3，直到表 user 的末尾循环结束。

驱动表全表扫描，被驱动表走索引树，然后回表进行匹配，符合的数据就直接作为结果集返回。

假设驱动表的行数是 N，执行过程就要扫描驱动表 N 行，然后对于每一行，到被驱动表上匹配一次(因为我们走的是索引)。

假设被驱动表的行数是 M。每次在被驱动表查一行数据，要先搜索索引，再搜索主键索引。每次搜索一棵树近似复杂度是以 2 为底的 M 的对数，记为 log2M，所以在被驱动表上查一行的时间复杂度是 2*log2M。

因此整个执行过程，近似复杂度是 N + N2log2M。

不走索引的join查询(Block Nested-Loop Join)

Simple Nested-Loop Join

还是上面的例子，如果把SQL改为如下，我们用没有索引的role_name去查询，执行计划会怎样呢？

这时候对于User表和role表走的都是全表扫描，那我们分析一下这句SQL。

先从user表中取出数据后，取role表中匹配，但这时候没有索引，所以我们的匹配是需要全表扫描role的。如果user表中有N行，role表中有M行，那我们匹配需要扫描的有N*M行！(扫描不是计算)这明显是不合适的。

以上的被称为Simple Nested-Loop Join，MySQL已经对这种join方式进行了优化，就算是不走索引的连表查询也是走的Block Nested-Loop Join。

Block Nested-Loop Join

对于Block Nested-Loop Join会使用线程内存join_buffer，其算法流程如下：

  1. 把表 user 的数据读入线程内存 join_buffer 中，由于我们这个语句中写的是 select *，因此是把整个表 user 放入了内存；
  2. 扫描表 role，把表 role 中的每一行取出来，跟 join_buffer 中的数据做对比，满足 join 条件的，作为结果集的一部分返回。

驱动表全表扫描放入 join_buffer 然后从被驱动表一行一行的拿过来匹配，符合的作为结果集返回。

由于使用了join_buffer，我们对user和role表做了一次全表扫描，因此扫描的行数是M+N；对表 role 中的每一行，都要做 M 次判断，总共需要在内存中做的判断次数是：M*N 次。相比于 Simple Nested-Loop Join 判断次数都是M * N ，但扫描行数少了是M + N。

join_buffer中放入的是驱动表，不会放入被驱动表，当我们去扫描被驱动表时，因为驱动表数据都在内存中，因此我们是取一条被驱动表的数据，在内存中匹配所有的驱动表数据，因此扫描的总行数是M+N，判断次数M*N。类似如下伪代码：

for(被驱动表){

​ for(驱动表){

​ 被驱动表第i条数据 匹配所有的驱动表数据

​ }

}

Join_buffer的大小是通过join_buffer_size设置的(默认值是 256k)，如果表很大，我们把驱动表和被驱动表放入join_buffer，极有可能出现放不下的问题。这时候的策略也很简单，分段放，这也就是Block的来源。执行顺序如下：

  1. 扫描驱动表，顺序读取数据行放入 join_buffer 中，当放到 i 行，放不下了，进入第二步；
  2. 扫描被驱动表，把被驱动表中的每一行取出来，跟 join_buffer 中的数据做对比，满足 join 条件的，作为结果集的一部分返回；
  3. 清空 join_buffer；
  4. 继续扫描驱动表，顺序读取剩下的数据放入 join_buffer 中，继续执行第 2 步。

这个时候，驱动表会多次分块放入join_buffer，假如分为 i 次，则被驱动表会被重复扫描 i 次。假设，驱动表的数据行数是 N，需要分 K 段才能完成算法流程，被驱动表的数据行数是 M。注意，这里的 K 不是常数，N 越大 K 就会越大，因此把 K 表示为λ* N，显然λ的取值范围是 (0,1)。

所以，在这个算法的执行过程中：扫描行数是 N+λ* N * M；内存判断 N * M 次。

该怎么选择驱动表

我们先来总结一下上面join的查询原理：

  1. 走索引的join查询

    对于被驱动表走索引的join查询，其查询原理是，驱动表全表扫描，被驱动表走索引树，然后回表进行匹配，符合的数据就直接作为结果集返回。

  2. 不走索引的join查询

    对于被驱动表不走索引的join查询，其查询MySQL会优化使用block Nested-Loop Join（查询计划extra 有block Nested-Loop Join，说明就是走的这种查询方式），block Nested-Loop Join算法的原理是，驱动表全表扫描放入 join_buffer 然后从被驱动表一行一行的拿过来匹配，符合的作为结果集返回；这里由于join_buffer大小有限，可能需要分块将驱动表数据放入，这样可能需要多次遍历被驱动表。

能不能使用 join 语句？

  1. 如果可以使用 Index Nested-Loop Join 算法，也就是说可以用上被驱动表上的索引，其实是没问题的；
  2. 如果使用 Block Nested-Loop Join 算法，扫描行数就会过多。尤其是在大表上的 join 操作，这样可能要扫描被驱动表很多次，会占用大量的系统资源。所以这种 join 尽量不要用。

所以你在判断要不要使用 join 语句时，就是看 explain 结果里面，Extra 字段里面有没有出现“Block Nested Loop”字样。

如果要使用 join，应该选择大表做驱动表还是选择小表做驱动表？

  1. 如果是 Index Nested-Loop Join 算法，应该选择小表做驱动表；因为驱动表走的全表扫描，被驱动表走索引匹配，扫描行数：N+N*2log2M（N 驱动表的总行数；M被驱动表的总行数）

  2. 如果是 Block Nested-Loop Join 算法：

    • 在 join_buffer_size 足够大的时候，是一样的；
    • 在 join_buffer_size 不够大的时候（这种情况更常见），应该选择小表做驱动表。所以，这个问题的结论就是，总是应该使用小表做驱动表。

    对于Block Nested-Loop Join 算法，不管用大表还是小表做驱动表，其匹配的次数是一定的都是N*M，但如果join_buffer_size不够大，也就需要多次放入join_buffer时，那么会对被驱动表扫描多次，这时候让分块的次数越少(也就是驱动表数量也少)，扫描的行数越少。

而对于什么是小表，我们需要在决定哪个表做驱动表的时候，应该是两个表按照各自的条件过滤，过滤完成之后，计算参与 join 的各个字段的总数据量，数据量小的那个表，就是“小表”，应该作为驱动表。

join优化

MMR

在介绍 join 语句的优化方案之前，先介绍一个知识点，即：Multi-Range Read 优化 (MRR)。这个优化的主要目的是尽量使用顺序读盘。

假如我们的查询语句如下，可以看到查询计划extra中有using mrr这就表示使用了mrr。

station_id加了索引，按照普通索引查询的方式，那么其执行过程如下：

  1. 从station_id索引树上查找station_id=1的主键；
  2. 根据找到的主键，去聚簇索引上查询数据(select *)，这里注意一次回表只能查找一条数据，然后返回结果集；
  3. 重复1，2，直到station_id=100；

上面的查询回表操作是随机读的，因为大多数的数据都是按照主键递增顺序插入得到的，而如果按照主键的递增顺序查询的话，对磁盘的读比较接近顺序读，能够提升读性能。

这就是MRR的优化思路，优化后的流程如下：

  1. 根据索引station_id，定位到满足条件的记录，将 id 值放入 read_rnd_buffer 中 ;
  2. 将 read_rnd_buffer 中的 id 进行递增排序；
  3. 排序后的 id 数组，依次到主键 id 索引中查记录，并作为结果返回。

这里，read_rnd_buffer 的大小是由 read_rnd_buffer_size 参数控制的。如果步骤 1 中，read_rnd_buffer 放满了，就会先执行完步骤 2 和 3，然后清空 read_rnd_buffer。之后继续找索引 station_id 的下个记录，并继续循环。

MRR 能够提升性能的核心在于，这条查询语句在索引 a 上做的是一个范围查询（也就是说，这是一个多值查询），可以得到足够多的主键 id。这样通过排序以后，再去主键索引查数据，才能体现出“顺序性”的优势。

BKA对于走索引join的优化

MySQL 在 5.6 版本后开始引入的 Batched Key Access(BKA) 算法了。这个 BKA 算法，其实就是对 NLJ 算法的优化。

NLJ 算法执行的逻辑是：从驱动表 user，一行行地取出 station_id 的值，再到被驱动表 role 去做 join。也就是说，对于表 role 来说，每次都是匹配一个值。这时，MRR 的优势就用不上了。

那怎么才能一次性地多传些值给表 role 呢？方法就是，从表 user 里一次性地多拿些行出来，一起传给表 role。既然如此，我们就把表 user 的数据取出来一部分，先放到一个临时内存。这个临时内存不是别人，就是 join_buffer。通过上一篇文章，我们知道 join_buffer 在 BNL 算法里的作用，是暂存驱动表的数据。但是在 NLJ 算法里并没有用。那么，我们刚好就可以复用 join_buffer 到 BKA 算法中。

使用BKA算法需要先开启MRR，那MRR在BKA算法中有什么作用呢？

先把驱动表表（小表）中查询需要的字段放入join_buffer, 然后把join_buffer里的字段值批量传给被驱动表，先根据索引查到id，然后得到一批主键id，再根据主键id排序，然后再根据排完序的id去主键索引查数据（这里用到MRR）

使用BKA需要先开启MRR，查询计划中有Batched Key Access代表使用了BKA算法。

不走索引join的优化

Block Nested-Loop Join 算法的性能问题

  1. 可能会多次扫描被驱动表，占用磁盘 IO 资源；
  2. 判断 join 条件需要执行 M*N 次对比（M、N 分别是两张表的行数），如果是大表就会占用非常多的 CPU 资源；
  3. 可能会导致 Buffer Pool 的热数据被淘汰，影响内存命中率。

优化

  1. 在被驱动表上建索引，走BKA算法

  2. 对于一些低频SQL，且不适合做索引的SQL，可以通过建临时表，并为临时表加索引的方式来优化查询。

  3. MySQL8之后开始支持hash join，我们也可以在业务代码中自己用hash join的方式来优化。

    把驱动表和被驱动表where筛选后的数据全部放入内存，用Hash来存放，然后遍历小表去匹配。
