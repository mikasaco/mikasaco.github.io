源码解读-类加载器
发表于 2019-11-02| 更新于 2019-11-02|源码解读类加载器

我们知道虚拟机的类加载机制的第一步就是加载（可以看我之前的博客），加载步骤需要完成3个功能：

  1. 通过类的全限定名获取这个类的二进制字节流；
  2. 将字节流代表的静态存储结构（static）转化为方法区的数据；
  3. 在方法区中生成一个class对象作为程序访问的入口。

java的类加载器的作用是将class字节码文件，加载进内存。本文就分析一下类加载的大致过程，和java内置的类加载器。主要分为2块：

  1. 对类加载器的父抽象类ClassLoader进行分析。
  2. 对java的几种类加载器进行分析

抽象类ClassLoader

对于所有的类加载器（Bootstrap ClassLoader除外）都需要继承这个抽象类ClassLoader，因此我们先从这个类开始分析，其中最重要的API就是加载类的方法 loadClass ,对于这个类的分析也将以这个方法为中心展开。

loadClass方法是抽象类ClassLoader定义的符合双亲委托机制的方法，这个方法里需要注意的有4点：

  1. 加锁的方法getClassLoadingLock
  2. 双亲委托机制的实现
  3. 加载类的操作findClass
  4. resolveClass关联class

可以看到parallelLockMap的初始化是建立在ParallelLoaders.isRegistered(this.getClass())==true的基础上的，这个ParallelLoaders是ClassLoader类里面的一个静态内部类，ParallelLoaders.isRegistered的判断是说这个类加载器是否有并行能力。

然后我们再看一下ConcurrentHashMap.putIfAbsent返回的是什么：putIfAbsent方法，如果传入key对应的value已经存在，就返回存在的value，不进行替换。如果不存在，就添加key和value，返回null

parallelLockMap中还没有key=className 新建的Object对象

双亲委托机制的实现

什么是双亲委派模型：当一个类加载器收到了类加载请求时，它首先不会自己去加载，而是交给父类去完成，每一层次的类加载器都这样，因此所有的类加载请求都会先传递给顶层的启动类加载器；当父加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。

为什么要用双亲委派机制：我们知道一个类的确定由类本身和类加载器共同决定，那这种设计简单的说，能够保证虚拟机里只有一个Object，java类型体系中最基础的行为都无法保证。

源码中如何实现：我们看ClassLoader.loadClass方法的获取类加载器的这部分代码片段，可以发现实现很简单，就是当加载类是，去判断parent != null，如果不为null那么就调用父类加载器，依次递归，直到顶层的类加载器bootstrap，去找bootstrap是否加载了这个类，如果加载了就返回类的class对象，没有返回null（这是在findBootstrapClassOrNull方法里的,看下面第二段代码）。如果bootstrap还是返回null，那么当前类加载器回自己去尝试加载类(if(c==null){c=findClass})。

加载类的操作findClass

我们看到在ClassLoader抽象类里，这个方法并没有具体的实现，我们阅读一下这个方法的注释，总结一下有两点：

这个方法具体实现由继承ClassLoader的类去重写。

这个方法会被loadClass调用。

在SecureClassLoader中并没有重写findClass方法，在URLClassLoader中重写了。

上面的核心代码很简单，就是得到了需要加载的类路径，然后最终调用defineClass方法，也就是说findClass方法的作用是找到类的路径。我们再跟着看一下defineClass方法的源码，发现就是通过jdk nio的方式去加载了class文件资源，读到了内存中，而最终是会去调用ClassLoader的defineClass方法，这里的作用是将读到内存的字节数组转换成class对象。

以上我们就理清楚了类加载的源码处理过程了,我们对loadClass、findClass、defineClass这三个方法的功能进行一个对比。

方法 功能 说明
loadClass 加载类 ClassLoader的方法，会调用findClass方法
findClass 查找类 找到类的资源路径，会调用defineClass方法，在URLClassLoader中重写了该方法
defineClass 将类读到内存，并转换成class对象 URLClassLoader中的defineClass方法通过nio的方式加载class文件资源，读到内存中，并会调用ClassLoader的defineClass方法；ClassLoader的defineClass方法将内存的字节数组转成class对象

第二点：findClass这个方法会被loadClass调用。虽然看着不起眼，但有层隐含的意思。

loadClass中包含了双亲委派机制的实现，而后面的加载是通过findClass去实现的。所以，如果我们自己实现一个类加载，而想保持双亲委派机制，就一般不重写loadClass，而去重写findClass；如果想破坏双亲委派机制，那么就重写loadClass。

resolveClass关联class

经过上面的步骤我们已经得到了需要加载的类的class对象（也就是说已经完成了加载了），那么就根据resolve的值决定是否调用resolveClass方法。

BootStrap

这个类加载器加载的是jre目录下的rt.jar、util.jar、charsets.jar等，像Integer、ClassLoader、String等都是这个加载的，此外需要注意的是，BootStrap是由C++语言写的，内置在jvm中，当JVM启动后，Bootstrap ClassLoader也随着启动，负责加载核心类库后，并构造Extension ClassLoader和App ClassLoader类加载器。我们在jdk源码中也找不到这个类，所以，如果我们获取class的类加载器，获取到的是null。

ExtClassLoader

这个类我们在==Launcher==中可以找到，其继承自URLClassLoader，加载位于$JAVA_HOME/jre/lib/ext目录下的扩展jar（如下图）。Launcher被称为java程序的入口(我没太明白这里)，其初始化的时候会创建ExtClassLoader和AppClassLoader。

[外链图片转存失败(img-4iZ2DrZp-1567309376095)(/Users/jerryq/Library/Application Support/typora-user-images/image-20190724000037574.png)]

AppClassLoader

继承自URLClassLoader，也叫系统类加载器（ClassLoader.getSystemClassLoader()可得到它），它负载加载应用的classpath下的类，也就是开发中自己定义的类都是由这个加载的。

还有一个需要注意的是，APPClassLoader重写了loadClass方法，看源码发现重写的loadClass方法只是增加了一些校验，最后还是调用了ClassLoader的loadClass方法，还是遵循双亲委派机制的。

线程上下文类加载器

在上面的Launcher构造方法中，还有注释3没讲，这里是设置了线程上下文类加载器，默认是AppClassLoader。线程上下文类加载器，可通过方法 getContextClassLoader() 和 setContextClassLoader() 获取和设置。在没有指定线程上下文加载器的情况下，线程将继承父线程的上下文类加载器。==Java 应用运行的初始线程的上下文类加载器是应用类加载器，所以在不指定的情况下就默认是 应用类加载器。==
