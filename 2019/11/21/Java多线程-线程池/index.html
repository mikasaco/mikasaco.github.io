Java多线程-线程池
发表于 2019-11-21| 更新于 2019-12-05|Java多线程线程池

一图概括

前言

线程池是多线程的重点，使用最多的方式就是直接使用ThreadPoolExector来构造线程池，这也是阿里规范推荐的做法，本文就重点分析ThreadPoolExector类的主要方法，以及一些关键的问题点：

  1. ThreadPoolExector构造方法的7个参数及其含义
  2. ctl变量以及线程池的5种状态
  3. 线程池怎么让核心线程存活的&超时时间怎么影响非核心线程的

构造方法

ThreadPoolExector的构造方法最终都会调用最长的那个构造方法，其中定义了7个参数，这几个参数经常在面试中问到，我们这里介绍一下这7个参数，以及对线程和任务的影响。

  1. corePoolSize—核心线程数

  2. maximumPoolSize—最大线程数

  3. keepAliveTime + unit — 存活时间和单位

  4. workQueue—任务队列

    任务队列是阻塞的队列，一般有3种：LinkedBlockingQueue无界队列，ArrayBlockingQueue有界队列和SynchronousQueue同步移交队列。无界队列用链表实现，可能会有链表很长，队列很长，响应时间太长，甚至引起oom，所以我们一般用有界队列。

  5. threadFactory—线程工厂

    一般用来设置线程的名字。

  6. handler—拒绝策略

    默认的拒绝策略是抛出异常(也就是这个时候再来任务之间丢异常了)，此外还有直接拒绝，丢弃最老的任务和返回让用户自己处理。

参数对任务处理的影响

这部分逻辑其实就是线程提交的逻辑，可以看下execute()方法。

  1. 线程池初始化之后，默认池中还没有线程。但也可以通过调用prestartCoreThread()方法，让线程池启动的时候就创建所有核心线程；
  2. 当池中线程数小于核心线程，此时来一个任务就创建一个核心线程。代码6～8行；
  3. 当核心线程满了之后，再来任务，会放入任务队列中，等待核心线程处理。代码11行；
  4. 当核心线程满了，且任务队列也满了，此时任务来了，就创建非核心线程去处理。代码18行；
  5. 如果池中有核心线程和非核心线程，而任务队列为空(任务都处理完了)，那默认对于非核心线程会存活 存活时间(keepAliveTime + unit)之后销毁，核心线程不会销毁，但如果将allowCoreThreadTimeOut变量设置为true，那么核心线程也会销毁；
  6. 当任务队列满了，且池中线程已经达到最大线程数了，那这个时候就走拒绝策略。代码18行。

主要流程是这样，但代码实现上还多了很多校验，例如校验线程池的状态。

这里我们也可以看到，其实线程池并没有区分核心线程和非核心线程，超过corePoolSIze创建的线程就是非核心线程。

线程池的5种状态

对于线程池也有不同的状态，不同的状态线程池能做的事也不一样。线程池有5种状态：

  1. new 线程池之后就是running状态；
  2. 调用shutdown()方法就进到shutdown状态；
  3. 调用shutdownNow()方法进到shop状态；
  4. 当线程池中任务队列为空，且没有工作线程就从shutdown/stop状态进到tidying状态；
  5. 进到tidying状态就会自动调用terminated()方法，然后进入terminated状态。

ctl变量

需要注意的是线程池的状态和线程池中的线程数是保存在一个AtomicInteger中的，就是变量ctl。

他将一个int 32位分成了两段，最高三位保存线程池状态(5种状态需要3位)，剩下的29位保存线程数(所以最大的线程数是2^29-1)。

那ctl是怎么实现一个int同时保存线程池的状态和线程池中的线程数的呢？

将5种线程池状态左移29位，这样这个int就是高3位存放了状态，后面当有新的线程创建时，自增ctl就可以了。

线程池如何做到核心线程一直存活的？

我们知道一个线程运行完成之后就会自己销毁，但线程池保证核心线程可以一直存活，这是怎么做到的呢？

线程提交之后ThreadPoolExector执行的流程

当我们调用ThreadPoolExector.execute(worker)之后，就把一个任务提交到了线程池，等待其处理，接下来ThreadPoolExector会依次调用

  1. execute()方法判断选择走哪一种情况(核心线程处理、非核心线程处理还是放入队列还是拒绝)；
  2. addWorker()方法创建线程，其中具体的创建线程操作是new Worker(),并调用Worker.start()方法来运行线程；
  3. Worker类执行run()方法，并调用getTask()方法从队列中取任务执行。

我们重点看下runWorker()方法的这部分代码。

这段代码的含义就是循环去从任务队列中取任务(getTask())，然后在处理完之后又把task置为null。而如果获取任务getTask()返回的是null，也就是任务队列已经为空了，那么就会跳出循环，这样这个线程就销毁了。

接下来getTask()获取任务的方法就是核心线程不销毁的原因了。

关键就在代码27～29行。这里BlockQueue.poll()是有超时时间的获取任务队列的任务，如果超时了那返回的任务就是null；而take()是会一直等待的获取任务。

addWorker()方法还有一个很有意思的使用，第一次看到Java 标签的使用，addWorker()定义了一个标签retry，相当于标记了下面的循环(addWorker()方法中是双重循环)，然后在循环中如果break retry直接就跳出循环往下走了，而continue retry会再次回到retry开始双重循环。
