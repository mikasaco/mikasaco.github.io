Java多线程-Java锁实现
发表于 2019-11-28| 更新于 2019-12-11|Java多线程锁

为什么会有Java锁

JVM提供的关键字synchronized在1.6版本之前使用的是传统的重量级锁，性能差，并且没有中断、公平锁等实现，所以在1.5的时候用Java语言实现了多种实用高效的锁，都在J.U.C包下。

而对于Java Lock最关键的是AQS抽象类，也叫做同步器，他封装了各种锁的底层实现，并用模版方法的设计模式给各种锁提供了易于实现的方式，只要子类重写其部分方法就可以很简单的实现锁。

但AQS并不能直接拿来当锁用，所以我们使用一般用其子类ReentrantLock等。

AQS

实现3要素

  • 状态变量：记录锁的状态(volatile修饰)，也可以认为就是锁资源，state==0 说明锁现在空闲可以拿来加锁，state>1说明是重入锁了

  • 队列：把等待锁的线程放到先进先出队列中，队列是用双向链表实现的 CLH队列

  • Unsafe类：CAS变更状态变量，阻塞和恢复线程(park/unpark)

AQS中队列分为两种：同步队列和等待队列。同步队列是给lock/unlock用的，相当于synchronized；等待队列是给await/signal用的，相当于Object.wait/notify。

模版方法设计模式

AQS使用了模版方法设计模式，在AQS类中实现了变更锁状态、入队出队、暂停恢复线程等操作，提供了锁的大部分处理，但在具体执行锁功能中的某些步骤，是让子类去重写的。

模版方法的设计模式，让AQS提供了大部分主要功能，但也提高了扩展性，例如独占锁、分享锁 我们只需要重写一部分方法就可以实现。

一般让子类重写的方法：独占/分享锁的获取锁和释放锁。

CLH队列

入队的节点放到双向链表的尾部，并让前一个节点的 waitstatus 设置为 signal ,然后自己进入阻塞状态；这样让前一个节点出队的时候就会唤醒当前节点。

节点出队会唤醒下一个节点，下一个节点唤醒之后要做的就是继续尝试获取锁资源。

ReentrantLock

类继承关系

ReentrantLock并不是AQS的子类，内部类Sync才是AQS的子类。ReentrantLock只实现了Lock接口。

ReentrantLock默认是非公平锁，ReentrantLock(true)时是公平锁。而公平锁和非公平锁的实现区别就是通过内部类NonfairSync和FairSync区别的。

lock/unlock

使用方式

模拟对共享数据的访问 static Integer i，加锁之后线程安全。

加锁实现

  1. 尝试获取锁资源，CAS 将 state 0改为1，成功的话就可以直接返回执行同步代码(这一步是只有非公平锁才有的)；
  2. 判断锁资源是否空闲(state==0)和队列是否为空，都满足的话就可以获得锁然后执行同步代码；
  3. 2的判断不满足，再判断当前持锁的线程是不是本线程，是的话state+1 实现锁重入；
  4. 2 3 都不满足，就需要把当前线程放入队列，放入队列需要考虑线程安全，这里需要使用 CAS 操作放入队列；
  5. 放入队列之后还会判断前面的节点是不是头节点，是的话就再尝试获取锁；
  6. 把前面节点的 waitstatus 置为 signal ，当前线程阻塞，当前面的节点出队的时候就唤醒后面的节点，然后后面的节点再去尝试获取锁资源。

解锁实现

  1. 锁资源变量 state -1，如果 state 减到0了，说明锁释放了；

  2. 取队列头部的第一个节点(不需要考虑线程安全)，把第一个节点的 waitstatus 改为 0 ，并且唤醒下一个节点。

公平锁和非公平锁的差异

公平锁和非公平锁唯一的差异就是在 非公平锁在加锁的第一步就会尝试去获取锁。后面的步骤都一样。

如果在获取锁的瞬间，正好锁资源空闲出来了，非公平锁就会立刻获取到锁资源，而不去队列中。所以如果队列里等待的线程一直被这么”插队”就会一直获取不到锁资源；但ReentrantLock默认用的就是非公平锁，也是因为这种方式会比公平锁更快，CPU唤醒线程更少。

非公平锁就是每次过来都老老实实的去队列排队，如果轮到他了就可以获取到锁资源。

可重入的实现

可重入的实现靠的是 state 锁资源这个变量：

  • state==0 ，当前没有线程持有锁资源；

  • state==1， 锁资源被某个线程持有；

  • state>1， 锁资源被某个线程重入了。

实现就是在获取锁的代码中，会判断当前线程是不是锁的持有线程，是的话 state+1 ，这就实现了锁重入。而在释放锁的时候也会让 state-1，当 state == 0 的时候就释放锁了。

可中断

ReentrantLock是可中断的，中断的意思是：线程获取锁等待了一段时间如果不想获取了，调用Thread.interrupt()可以中断等待。而对于synchronized是不能做到这个的，线程等待获取锁，只能一直等待下去。

await/signal

AQS中除了实现了类似synchronized的锁，也有类似Object.wait/notify线程通信机制的await/signal。在AQS中还有一个内部类 ConditionObject ，这个就是等待唤醒通信实现的类。

实现 await/signal 也是依靠3个部件：锁资源(变量 state)、队列、Unsafe的CAS操作和park/unpark。其中需要注意的是，这里的队列不同于 lock/unlock 的同步队列，而是在 ConditionObject 类内部定义的 等待队列。此外，同步队列头节点虚拟节点，但等待队列头节点会存储数据。

使用方式

根据上面的demo和返回结果有几点注意事项：

  1. 使用 await/signal 之前都需要调用 lock 获取锁，跟 wait/notify调用都需要在synchronized代码块/方法中一样，因为需要等待的锁对象是什么；
  2. Thread 2 唤醒 thread 1 不是在 signal 之后，而是需要 thread 2 释放锁之后。

await/signal交互

  1. 调用线程必须先获取到锁资源，在当调用 lock.await() 后，放入等待队列，阻塞调用线程，释放锁资源；
  2. 另一线程也必须先获取锁资源，在当调用 lock.signal() 后，把等待队列的头节点移到同步队列，signal()方法只是做了这一点；
  3. 另一线程在执行 lock.unlock() 释放锁资源之后，才会唤醒之前 await 的线程。

其实这里的3步骤唤醒等待的线程，走的还是之前 lock/unlock 的方式，其实是让这个等待线程自己排队去获取锁资源了。

注意事项

调用之前需要先获取锁

无论是 await 还是 signal 调用之前都需要先获取锁对象，否则会抛出 非法监视器状态异常 IllegalMonitorStateException。

其中 await 是在释放锁的时候抛出的， signal 是在第一步就会判断当前锁持有线程是不是本线程。

signal可以单独出现

signal 方法的作用只是把 等待队列中的头节点移除，放入同步队列，如果等待队列为空就不移除，也不会有问题。

await之后是会释放锁资源

调用 await 之后会释放全部的锁资源，这是为什么需要调用之前先获取锁，也是 siganl 线程可以获取锁 的原因。

唤醒等待线程是在释放锁之后

signal的作用单单是把等待队列头节点放到同步队列，并不会唤醒等待线程，等待线程后面就是跟 lock/unlock 一样去获取锁资源然后唤醒。

如果是公平锁的同步队列，那这个等待线程就排队去获取锁资源唤醒，如果是非公平锁还可能被插队。
