MySQL-一主多从集群
发表于 2019-11-15| 更新于 2019-11-15|MySQL主从集群

一图概括

前言

在生产业务系统中，MySQL为了保证高可用和高并发一般都会采用一主多从的集群架构，架构图如下图所示。A是主库，A‘是备库，这种一主一备的设计保证了高可用(A挂了 A‘能顶上去做主库)；而B C D(包括A‘)都是只读的，往往业务系统都是读多写少，这样可以分摊读的压力，支持高并发。

主库和备库、主库和从库需要保障数据的一致性，我们都知道是基于binlog实现的，但具体是怎么做的呢？ binlog具体内容是什么？ 本文就从binlog开始分析一下MySQL一主多从的架构设计。

备库和主库是双M关系，备库会在主库挂了的时候当主库，备库的存在是保障高可用；

从库和主库是为了读写分离，保证高并发下的读写能力

binlog的内容格式

binlog分为3种模式，statement、row和mixed，在做主从建设中我们一般都选择row模式，为什么呢？要明白这个问题，其实很简单，看下这三种binlog日志的信息以及可能出现的问题就可以了。

statement

statement模式记录的是SQL语句。但statement模式可能出现主备不一致的问题：

case1：如果SQL中有 now()函数，那么主库和从库获取当前时间肯定是不一样的(主备同步延迟)，这就造成了主备数据不一致。

case2：例如SQL中delete带limit就很有可能出现主备删的数据不一样导致数据不一致。因为可能主备库走的索引不一样。

row

row模式中记录的是哪行数据改了什么，有改之前的数据行和修改的操作是什么，改之后是什么信息，也因为记录了修改之前的数据，可以用来恢复误操作。

row方式不会出现主备不一致的情况，但数据量大，比如修改一万行数据，statement模式只需要记录一条SQL，row模式会有一万条日志。

statement模式使用new()函数出现主备时间数据不一致的问题，row模式binlog里会有时间戳，用这种方式避免了主备使用new()函数，数据不一致的问题。

mixed

mixed模式是MySQL 自己会判断这条 SQL 语句是否可能引起主备不一致，如果有可能，就用 row 格式，否则就用 statement 格式。但有可能选错。

同步流程

同步流程如下图所示。

对于主库，会启动一个 dump_thread线程将binlog日志发送给从库；

从库会启动io_thread线程来接收binlog日志，并将其转为中转日志(relaylog)；

从库还会有一个sql_thread线程将中转日志解析成相应的数据。

主备模式

MySQL主备模式是为了实现高可用设计的，一般会采用双M的架构，也就是两台实例互为主备。

循环复制问题

如果数据库A中插入一条数据，那么数据库B作为备库会接收这个插入数据的binlog，然后在备库B中也插入这条数据，但现在双M架构，AB互为主备，B在插入这条数据之后又会把binlog日志同步给数据库A，如果这时候数据库A把这条binlog日志再转成数据存下来了，那么就会多了一条数据，这就是循环复制问题。

MySQL解决这个问题很简单，MySQL配置文件会指定自己实例的service id，如果数据库A收到的binlog日志是自己的(service id是自己的id)，那么就丢弃了这条日志，这样就避免了循环复制的问题。

这就是为什么MySQL做主备、canal同步数据，都规定service id不能一样的原因。

主备延迟问题

主备延迟就是同一个事务在主库和从库完成的时间差。这其中包括主库同步到从库，从库解析日志以及执行日志的时间。

由于有主备延迟问题的存在，会有两种不同的主备切换策略：可靠性优先和可用性优先。

可靠性优先策略能保证不会出现数据不一致的问题，但会有一段时间的集群可不用；

可用性优先策略不会存在集群不可用，但可能会出现数据不一致的问题。

可靠性优先的切换策略：判断主备的延迟时间小于某个阈值(例如5秒)，然后把主库设为只读，直到延迟时间为0后，将从库设为读写，这样就切换了主备。

可用性优先的切换策略：直接让主库备库都是读写状态，再等延迟时间为0后，把原来的主库设为只读。这两个库都可以写入，保证了可用性，但可能出现数据不一致的情况。

主从模式

一主多从的结构，其实就是读写分离的基本结构，读写分离的主要目标就是分摊主库的压力。但由于主从可能存在延迟，客户端执行完一个更新事务后马上发起查询，如果查询选择的是从库的话，就有可能读到刚刚的事务更新之前的状态。

为了解决读写分离可能出现从库读的数据由于延迟问题和主库不一致的问题，我们一般有以下这些解决方案：

  1. 强制让部分读走主库。
  2. 读从库的时候延时一段时间。
