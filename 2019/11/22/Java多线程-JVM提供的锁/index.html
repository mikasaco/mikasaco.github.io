多线程-JVM提供的锁
发表于 2019-11-22| 更新于 2019-12-11|Java多线程锁

前言

首先，Java内存模型(JMM)千万不要跟JVM结构搞混了，这两者是不同层次的概念。

首先如果别人问什么是JMM，我们该怎么回答？

Java内存模型是在计算机模型基础上设计的，能够屏蔽操作系统和硬件差异的；产生的目的和计算机模型一样，都是为了解决多线程操作共享数据带来的数据不一致问题；然后Java内存模型分为两个部分：模型设计和操作规范，模型设计主要就是主内存、工作内存和线程的关系，操作规范有8种，我们一般用先行发生原则来判断。

那可能还会接着问，什么是计算机模型？具体怎么操作共享数据的？并发问题有哪些？

本文就从这开始展开，了解一下Java内存模型。对于JMM我们除了要知道他关键的模型外，类比计算机硬件的内存模型，了解JMM产生的来龙去脉，会帮助我们更好地理解JMM。

Java内存模型

  1. Java内存模型是在计算机内存模型上发展来的，JMM屏蔽了硬件区别和操作系统；
  2. 设计的目的我认为是为多线程操作共享数据提供一个统一的模型方案；
  3. Java内存模型可以分为两个方面：内存模型抽象结构(主内存和工作内存)和内存交互操作规范(也就是先行发生原则)。

JMM抽象结构

分为线程、工作内存和主内存，主内存保存数据，线程需要访问主内存中的数据时，要先将主内存的数据拷贝一份到工作内存，然后如果修改了工作内存的这个数据，还需要再同步回主内存。

JMM带来的数据不一致问题

如果多个线程访问了主内存中的共享数据，并都在各自的工作内存中进行修改，那此时就有工作内存和主内存数据不一致的问题。

哪些是共享数据呢？

在堆中的实例对象、静态对象等是共享数据。在栈中的数据 局部变量、方法参数等是线程私有的，不是共享数据。

JMM内存交互操作规范(happens-before)

JMM 的内存交互操作就是主内存和工作内存的交互。

JMM 内存交互操作包括8种原子操作和这些原子操作必须满足的规定。

8种原子操作：lock unlock read load write 等；操作必须满足的规则例如 不允许read load store write单独出现等。

这些原子操作的规定不好记，一般我们用happens-before原则判断。

Happens-before原则就是操作A如果先行发生于操作B，那么A的数据对B来说是可见的。

指令重排

指令重排是为了提高性能，会对没有依赖关系的指令重新排序。

多线程的数据安全

Java内存模型定义了多线程操作共享数据的方式，但JMM 是可能出现并发安全问题的，并发安全主要围绕着3个点。

原子性 synchronized

原子性就是说一段操作一旦开始了，就不能被中断，不会被其他线程打断。

synchronized底层使用lock/unlock保证中间操作的原子性。

volatile不保证原子性

可见性 synchronized volatile final

可见性是指当一个线程修改了共享变量的值，其它线程能立即感知到这种变化。

Java中volatile、synchronized和final三个关键字：

  • volatile的可见性是由于其修饰的变量在线程每次更新了之后回立刻同步回主内存，并且线程每次操作其修饰的变量也都需要去主内存读入(即使工作内存中已经有了)；
  • synchronized的可见性是由于在解锁unlock之前会把共享数据同步回主内存；
  • final的可见性应该就是由于其修饰的变量的不可变。

有序性 synchronized volatile

有序性是指线程内的操作都是有序的。

Java中主要是用volatile或synchronized两个关键字保证有序性。

  • synchronized是同一时刻只会有一个线程操作数据，所以有序；
  • volatile禁止指令重排保证有序。

volatile

volatile保证了有序性和可见性，其实现的原理是因为内存屏障。

volatile反编译之后会使用内存屏障，内存屏障就是一些特殊的指令(Lock前缀)。

  1. 内存屏障会让两侧的指令不能重排序；
  2. 将高速缓存中的数据强制写回内存。

第一点保证了禁止指令重排，进而保证了有序性；第二点保证了可见性。

对于第二点的说明：

高速缓存中的数据写回内存，也就是volatile修饰的数据在工作内存更新之后会把新值立刻同步到主内存，这样别的工作内存中的数据副本已经过期了，在使用的时候需要再从主内存中获取。

synchronized

synchronized保证了原子性、可见性、有序性，是线程安全的。

使用方式

底层实现原理

对于代码块和方法实现细节有差异，但JVM底层实现是一样的。

代码块是通过monitorenter/monitorexit指令，方法是会在生成了一个ACC_SYNCHRONIZED关键字。

Synchronized —— JVM monitorenter/monitorexit指令 —— JMM lock/unlock

对象头存储锁信息

锁信息是存在对象头的 mark word 中的，不同的锁其mark word 的信息不同。

  1. 偏向锁。线程ID、是否偏向、锁标志；
  2. 轻量级锁。指向栈的指针、锁标志；
  3. 重量级锁。指向堆中的指针，锁标志。

锁优化

JDK1.6之前的synchronized就是重量级锁，在1.6之后进行了优化，有了偏向锁和轻量锁。

偏向锁

很多的同步代码都还是被一个线程使用，其实并没有竞争，这时候频繁加锁解锁，性能很差。所以偏向锁就是锁对象一直被同一个线程访问时，只会在该线程第一次获取锁的时候加锁，并在有别的线程竞争的时候解锁。

轻量锁

偏向锁被多个线程竞争的时候就升级成轻量锁。这时候获取到轻量锁的线程执行同步代码，没有获取到轻量锁的线程不断自旋去尝试获取轻量锁，这时候线程不阻塞。

重量锁

当线程自旋了多次还是没有获取到轻量锁对象时，这时候轻量锁就会升级成重量锁。重量锁是需要切换用户态和内核态，线程会阻塞，性能差。

synchronized锁的特性

可重入锁是指共享变量的锁可以被一个线程多次加锁，而不会造成线程自己死锁了自己的情况。

前面说到synchronized反编译之后时monitorenter 和 monitorexit 指令，这两个指令的工作方式是怎么样的呢？

monitorenter 和 monitorexit 指令的工作方式：

monitorenter：当前线程尝试去获取共享变量的锁，如果该共享变量没有被锁住，或者当前线程已经有这个共享变量的锁了，那么计数+1；

monitorexit：当前线程释放锁，计数-1，当计数=0时，这个共享变量就释放锁了，可以被别的线程操作了

从上面的描述可以看到，synchronized符合可重入锁的特性。

非公平锁是说当共享变量的锁释放之后，后面获取到锁的线程是随机的，不是按照线程排队的顺序的。与之相对的是公平锁。

synchronized是非公平锁。

计算机内存模型

Java内存模型是在计算机硬件模型基础上发展来的，有很多相似的地方，这里也介绍一下计算机内存模型。

我们知道计算机数据存储在内存中(这里不讲磁盘)，而数据处理是在处理器中处理的。在处理器和内存之间还有高速缓存，一般分为3级：L1，L2，L3，三者大小依次递增，距离处理器越来越远，性能也依次递减，其中一般L1和L2是每个处理器独有的，L3是多核处理器共有的。下图就是计算机内存模型。

高速缓存cache

cache的出现是因为，处理器内核的处理速度远高于从内存读取数据的速度。

读数据

计算机读取数据的方式是先从cache一级一级往下读，最后再从内存读，内存读入之后回放到cache。

改数据

把主内存中的数据复制一个到cache进来，然后给到处理器去操作，再在之后同步给主内存，完成数据更新。

而CPU从内存读数据和从磁盘读数据一样都是以块为单位的。从内存读入的数据大小一般是64字节，这就是缓存行。CPU这种读数据的方式对数组有利，读一个数据会把相邻的数据也读入cache，下次访问更快。

缓存设计带来的问题

利用cache改数据时，对于多核CPU数据，可能有多个不同的副本和内存数据不一样，这时就发生了缓存不一致的问题。
