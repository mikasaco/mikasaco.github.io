MySQL-绕不开的索引
发表于 2019-11-06| 更新于 2019-12-04|MySQL索引

一图概括

前言

对于数据库而言，索引是查询性能优化的重点，很多慢查询都可以通过索引来优化，本文介绍一下索引的类型和索引的工作原理，以及使用索引的优劣。

索引的数据结构

MySQL索引的数据结构

在介绍MySQL的索引之前，先问一下自己，什么是索引？索引的目的是为了更快的查找，其实哈希表、有序数组、二叉树等都可以用来做索引的数据结构。

  1. 哈希表查询，没有哈希冲突的情况下，时间复杂度为O(1)，适合等值查询。

  2. 而有序数组的查询，可以用二分查找，时间复杂度O(log2N)，并且对于范围查询，只要移动指针即可，适合等值查询和范围查询，但是插入删除很麻烦，需要移动数据。

  3. 二叉树的查询效率也很高，查询时间复杂度O(log2N)。

  4. B树和B+树，是m阶树，数据库一般都采用这种数据结构。

我们都知道InnoDB(MySQL索引是在存储引擎层实现的，但主要都用InnoDB引擎)索引用的是B+树，那为什么不选用上述的这些数据结构呢？

对于像哈希表、有序数组这种就不必多说，我们就说一下树结构，二叉树、B树和B+树。

对于二叉树，每个节点只能存储两个节点，这样就会导致树的高度很高，而我们查询数据需要进行IO操作，树越高意味着我们需要更多次的IO操作，所以首先pass了。那为什么不用B树而是B+树呢？在讨论这个之前，我们先看一下什么是B树，什么是B+树。

几种树结构的对比

二叉树

二叉树是只有两个节点，并且左节点<父节点<右节点。但有可能会造成树倾斜在一边，这样查询效率低。

二叉查找树

二叉查找树和二叉树的不同在于，二叉查找树子树高度不超过1，这样保证了平衡，查询效率高。但在插入删除节点的时候可能会造成不平衡的情况，这时候就需要对二叉树进行维护调整。

B树

以上两种都只适合在内存中，最主要的原因是节点只能有两个，对于存储在外存中的数据，这样的查找方式会有太多次IO操作。

B树每个节点包含数据、指针、关键字三个部分，每个节点对应着一个外存中的数据块，其中数据是指真实需要的数据，指针是指向子节点的指针，关键字是查找对象的比较目标。需要注意的是每个节点查找都需要读入内存，需要一次io操作，然后对于读入内存的数据再进行内存中的查找方法，例如顺序查找或二分查找。B树的结构如下图所示。

B+树

b+树是在b树的基础上改进，更适合外存查询。B树中每个节点保存了数据、关键字和指针，当数据较大时每个节点能保存的关键字很少，整个结构也会变得更庞大，查询效率不高，为此b+树中只在叶子节点保存了数据，非叶子节点都只包含关键字和指针，这种方式使得每个非叶子节点能包含更多的关键字信息，减少io次数。

B+树和B树的区别

相同点：

B树和B+树都是M阶树，并且在每个节点中数据都是有序的，所以在树查找和节点中查找可以用二分查找的方式；而M阶树的数据结构相比于二叉树更适合MySQL，原因之一是之前提到的二叉树树高度太高，需要多次IO操作；其二是InnoDB 的数据是按数据页为单位来读写的，更适合MySQL数据查询。

不同点：

  1. B+树数据都存在叶子结点中，中间节点不存储数据，只存储索引，B树数据会存在所有节点中。B+树这样的设计，有两个好处：

    A. 会使得每个中间节点可以存储的索引更多，树的高度就相应低了，减少了IO操作；

    B. 数据都在叶子结点中，每次查询的高度相等(B+树中所有叶子结点都位于同一层)，查询性能很稳定，而B树可能查询结果在叶子结点也可能在中间节点，查询不稳定。

  2. B+树中所有的叶子结点会有指向下一个叶子结点的指针，这样可以遍历叶子结点。

索引的维护也是一个大问题，对于插入、删除等操作，往往会有节点分裂、合并等

InnoDB的索引模型

聚簇索引和二级索引

在 InnoDB 中，表都是根据主键顺序以索引的形式存放的，这种存储方式的表称为索引组织表。又因为前面我们提到的，InnoDB 使用了 B+ 树索引模型，所以数据都是存储在 B+ 树中的。因为无法把数据行存放在两个不同的地方，所以一个表只能有一个聚簇索引。

每一个索引在 InnoDB 里面对应一棵 B+ 树。下图左是主键的B+树，其叶子结点存储的整条数据；下图右是字段K的B+树，其叶子结点只存储了主键ID的数据。

主键索引的叶子节点存的是整行数据。在 InnoDB 里，主键索引也被称为聚簇索引（clustered index）。

非主键索引的叶子节点内容是主键的值。在 InnoDB 里，非主键索引也被称为二级索引（secondary index）。

聚簇索引和二级索引的区别

如果我们想查一条数据，如果通过主键ID查询，那么直接走聚簇索引就可以查询得到；如果通过二级索引查询，需要先在二级索引中查询到主键ID，再到聚簇索引中查询数据(回表)。

也就是说，基于非主键索引的查询需要多扫描一棵索引树。因此，我们在应用中应该尽量使用主键查询。

几种索引的概念

前面说了聚簇索引和二级索引，再来说说其他一些索引的概念。

覆盖索引

如果我们查询select id from T where a=1，并且字段a上建了索引，那么我们查询的id其实在a这颗索引树上就有，这样就不必再回表了，这就是覆盖索引。覆盖索引可以减少树的搜索次数，显著提升查询性能。

联合索引

上图我们建立了姓名和年龄的联合索引，这样做对于这句SQLselect age where name='李四' 就直接在联合索引树上就找到了，不需要在回表了。

最左前缀原则

上面联合索引中可以看到，索引排序是按照先联合索引的第一个字段，再是第二个字段这样排序的。利用这个特性，我们建立了(name,age)联合索引，也就不需要再建name索引了。

mysql 建立多列索引（联合索引）有最左前缀的原则，即最左优先，如：

如果有一个 2 列的索引 (col1, col2)，则已经对 (col1)、(col1, col2) 上建立了索引；
如果有一个 3 列索引 (col1, col2, col3)，则已经对 (col1)、(col1, col2)、(col1, col2, col3) 上建立了索引。

这里的理解也很简单，如果我们建了(name,age)的联合索引，那么除了这个联合索引，对于name我们已经排好序了，但对于age，他在name相等的时候是有序的，但从整个联合索引上来看，age并不是有序的。

唯一索引和普通索引

唯一索引并不等于聚簇索引，聚簇索引一张表只能有一个(因为聚簇索引是要存数据的)，唯一索引可以有多个。唯一索引只是保证索引键的唯一性，两者其实并不是一个概念。

而唯一索引和普通索引其实是比较类似的，两者唯一的不同是唯一索引需要保证索引字段的唯一性。而这个特性，又会为两者的查询效果带来较大的影响。主要体现在查询和更新两个方面。

查询区别

对于普通索引来说，查找到满足条件的第一个记录后，需要查找下一个记录，直到碰到第一个不满足条件的记录。

对于唯一索引来说，由于索引定义了唯一性，查找到第一个满足条件的记录后，就会停止继续检索。

这两者查询效果差异其实是微乎其微的，因为InnoDB 的数据是按数据页为单位来读写的，所以对于普通索引去查找下面不符合的记录的操作，只是在内存中移动一下指针。但也有可能正好数据在数据页的末尾，这样需要读下一块数据页，但概率很小。

更新区别

为了说明普通索引和唯一索引对更新语句性能的影响这个问题，需要先跟你介绍一下 change buffer。

如果update的数据页在内存中，那么直接更新这个数据页，然后在适当的时候写回磁盘；如果update的数据页不在内存中，MySQL会把更新的操作先写在change buffer中，在下次查询这个数据页的时候再把change buffer上的更新操作写到数据页上，或者后台会定时刷到磁盘上。

说明了change buffer的概念之后，需要知道唯一索引是不会用到change buffer的。因为唯一索引需要校验索引的唯一性，那如果数据页不在内存中，就需要从磁盘上读入内存，再判断唯一性；如果数据页在内存，那也没有用change buffer的必要了，直接更新内存中的数据页就好了。

唯一索引和普通索引在更新时，数据页不在内存中的情况，性能会有较大差异，唯一索引因为要校验索引的唯一性，必须读取数据判断有没有冲突；而普通索引直接把更新写到change buffer就结束了，少了一次IO操作。

所以通常一次更新update操作，不会立刻作用于磁盘。

前缀索引

可以定义字符串的一部分作为索引，这就是前缀索引。这样做的好处是减少单个索引键的大小，节点能够存储更多的键，查询效率更高。

索引维护

B+ 树为了维护索引有序性，在插入新值的时候需要做必要的维护。插入新值是，一个数据页满了，按照B+Tree算法，新增加一个数据页，叫做页分裂，会导致性能下降。空间利用率降低大概50%。当相邻的两个数据页利用率很低的时候会做数据页合并，合并的过程是分裂过程的逆过程。

索引的选择

索引可以加快查询，但会占用空间，此外维护索引(B+树)的成本也不低，所以对什么字段建索引也是需要注意的。

  1. 频繁更新的字段不适合建索引，索引的维护。

  2. 字段区分度很低的字段(性别)，不适合建索引。

  3. 字段长度很大不适合建索引，单个节点容纳的索引键少。
