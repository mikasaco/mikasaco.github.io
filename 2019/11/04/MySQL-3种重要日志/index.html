MySQL-三种重要的日志
发表于 2019-11-04| 更新于 2019-12-04|MySQL日志

一图概括

前言

MySQL有很多日志：binlog、redo log、undo log、慢查询日志等等。本文介绍binlog、redo log、undo log这三种重要的日志，以及他们在数据更新、服务宕机、数据库恢复等的作用。

WAL

再介绍这三个日志之前，先介绍一下MySQL一个重要的技术：WAL(Write-Ahead Logging)，它的关键点就是先写日志，再写磁盘。

具体来说，当有一条记录需要更新的时候，InnoDB 引擎就会先把记录写到 redo log里面，并更新内存，这个时候更新就算完成了。同时，InnoDB 引擎会在适当的时候，将这个操作记录更新到磁盘里面。

undo log(回滚日志)

含义

更新数据之后就会写入一条回滚日志，这里的更新包括update/delete/insert，而回滚日志中记录的是反操作。每个数据行中都有几个隐藏列，其中就有指向回滚日志的指针。

作用

  1. 事务的原子性需要回滚日志来保证，提供回滚操作。
  2. InnoDB事务的隔离性中，可重复读和读提交基于MVCC实现，而MVCC的实现也依赖回滚日志。

redo log(重做日志)

含义

数据更新之后，会将更新的内容写到重做日志中，重做日志分为两部分：内存中的 redo log buffer和磁盘上的redo log file。重做日志采用的是循环写的方式，大小是固定的，而写重做日志的时候是顺序写的，比直接更改索引组织表的数据更快，系统会在空闲或日志写满等情况下，将更改同步到数据库中(同步的时候是从内存buffer pool同步过去的，跟重做日志没有关系)。

作用

  1. 服务崩溃的时候，就是根据重做日志来恢复数据。
  2. 重做日志保证了事务的持久性。

redo log buffer

重做日志不是直接写入硬盘，是先写入 redo log buffer的。如果一个事务中有多个更新SQL，那么InnoDB会把每个更新SQL相应的重做日志放在redo log buffer中，在最后事务提交的时候，一起落到磁盘上的重做日志中。

需要将innodb_flush_log_at_trx_commit 这个参数设置成 1 ，这样每次事务提交时才会把redo log buffer 持久化到磁盘。

数据更新的最终落盘是从 buffer pool 更新过去的

在重做日志中记录的是”在某个数据页上修改了什么”，注意是数据页，所以直接从重做日志去更新数据是不能实现的。实际上数据的最终落盘是这样的：

  • 如果是正常运行的实例的话，数据页被修改以后，跟磁盘的数据页不一致，称为脏页。最终数据落盘，就是把内存中的数据页写盘。这个过程，甚至与重做日志毫无关系。
  • 如果崩溃之后重启实例，InnoDB会根据重做日志将对应的磁盘数据页读入内存放入buffer pool，然后再根据重做日志的内容修改内存中的数据页，数据页成了脏页之后又会再走正常实例的流程了。

InnoDB用buffer pool缓冲池来管理内存，内存中存的是数据页，数据页会有3种不同的情况：未使用的；和磁盘上数据页一致的干净页；和磁盘上数据页不一致的脏页

数据落盘的时机

  1. 重做日志大小固定，循环写，如果重做日志要写满了，就需要把buffer pool更改的数据页(脏页)刷到磁盘上；
  2. buffer pool满了，需要把部分脏页刷到磁盘；
  3. MySQL正常关闭；
  4. 系统空闲；

对于一个长时间运行的MySQL实例，其缓冲池常态应该是大部分都使用了的，所以缓冲池满了刷脏页落盘的情况是很多的。

当查询需要读入一个缓冲池中没有的数据页时，InnoDB就需要将这个数据页读入缓冲池，如果此时缓冲池满了，那么会遵循LRU算法淘汰一个最久没用的数据页。

如果这个最久没用的数据页是干净页，那么直接拿来用；但如果是脏页就需要刷盘了。

binlog(归档日志)

归档日志和重做日志的区别

MySQL service层提供的归档日志也会把每次操作都记录下来，但归档日志和重做日志有很大的不同的：

  1. redo log 是 InnoDB 引擎特有的；binlog 是 MySQL 的 Server 层实现的，所有引擎都可以使用。
  2. redo log 是物理日志，记录的是“在某个数据页上做了什么修改”；binlog 是逻辑日志，记录的是这个语句的原始逻辑，比如“给 ID=2 这一行的 c 字段加 1 ”。
  3. redo log 是循环写的，空间固定会用完；binlog 是可以追加写入的。“追加写”是指 binlog 文件写到一定大小后会切换到下一个，并不会覆盖以前的日志。

作用

归档日志最主要的两个使用场景：备份和主从。

如果在这个事务执行过程中服务崩溃了，对于前面的这些内存操作其实并没有持久化数据没有关系，我们看下持久化日志的几种崩溃时间点的情况：

  1. 回滚日志写完之后崩溃了。回滚日志写的是反操作，是数据原来的情况，所以没关系；
  2. 重做日志从redo log buffer落盘后崩溃了。此时binlog还没写，redo log也还没提交，恢复后会回滚；
  3. 写了binlog后崩溃了。此时恢复的时候会根据重做日志，如果redo log有commit标志，也就是事务提交完成了；如果redo log没用commit标志，那么看对应的binlog是否存在，存在的话，提交事务；不存在的话回滚事务。
