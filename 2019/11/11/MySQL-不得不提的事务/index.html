MySQL-不得不提的事务
发表于 2019-11-11| 更新于 2019-11-13|MySQL事务

一图概括

前言

MySQL的事务并不是天然支持的，MyIsam引擎就不支持事务，InnoDB支持，事务是由存储引擎实现的。事务在日常开发中也是相当重要的，面试中常问道事务的特性，如果只知道ACID就太伤了，本文就重点分析ACID的具体含义以及在InnoDB中的实现方式。

此外需要知道的是，如果没有显示使用事务(start transaction / commit)，那么每个 sql 语句都会被当做一个事务执行提交操作。

原子性

含义

原子性的含义是说，一个事务是一个不可分割的工作单位，事务的操作要么整体成功，要么整体失败，如果失败，那么就需要回滚。

实现原理

事务原子性的实现是依靠undo log回滚日志实现的。对于每个update/insert等操作，都会生成相应的undo log，回滚操作就是基于这些undo log实现的。

Undo log中记录是相反的操作。

持久性

含义

持久性说的是，事务执行完成之后，对数据库对改变是持久的，数据不会因为宕机等情况被影响。

实现原理

如果每次更新都去写磁盘，那么IO都需要翻账本(找数据改数据)，性能低下，类似掌柜记账那个例子。MySQL为了避免这种情况，每次更新并不是直接修改磁盘数据，而是先将更改的操作记录在redo log中，redo log虽然也是保存在磁盘上，但是顺序写的，性能较高，然后再在系统空闲(也可能redo log写满等情况)时把更改再写到磁盘，这种先写日志在写文件的操作也就是WAL。

当事务提交后，数据其实还未写到磁盘，这时候如果发生服务崩溃，我们还是能够通过redo log来恢复，InnoDB也就是通过redo log来保证事务的持久性。

innodb_flush_log_at_trx_commit 这个参数设置成 1 的时候，表示每次事务的 redo log 都直接持久化到磁盘，注意是事务完成之后写redo log，然后这个参数设置为1，每次事务完成刷到磁盘。

隔离性

含义

前两者主要研究的是单个事务本身的特性，而隔离性说的是多个事务之间的影响。

隔离性是事务之间相互隔离，不会受到别的事务的影响。

那如果没有隔离性，事务会受到什么样的影响呢？这里其实分为两部分，写写操作和写读操作。

InnoDB中事务隔离的实现

写写—写锁

两个事务的写操作的隔离是通过锁实现的。锁机制的基本原理可以概括为：

事务在修改数据之前，需要先获得相应的锁。

获得锁之后，事务便可以修改数据。

该事务操作期间，这部分数据是锁定的，其他事务如果需要修改数据，需要等待当前事务提交或回滚后释放锁。

锁按照粒度分为全局锁、表锁和行锁。MySQL 中不同的存储引擎支持的锁是不一样的，例如 MyIsam 只支持表锁，而 InnoDB 同时支持表锁和行锁，且出于性能考虑，绝大多数情况下使用的都是行锁。

写读—MVCC

异常读的情况

如果一个事务写数据，一个事务读这部分数据，根据不同的情况会有脏读、不可重复读和幻读这三种情况。

脏读

脏读就是事务读到了另一个事务还未提交的数据。

事务A读到了事务B还没有提交的数据(200)。

不可重复读

不可重复读就是一个事务中两次读一个数据，读到的值不一样。(另一个事务提交了的值)

在事务A中，T2和T5两次读zhangsan的余额，值不相同。

幻读

幻读就是一个事务中，两次读的条数不一样。(和不可重复读的区别：不可重复读是数据变了，幻读是数据的行数变了)

SQL标准的4种事务隔离级别

那真对上面这三种异常读的情况，SQL标准制定了4种隔离级别：读未提交、读已提交、可重复读、串行化。

这4种隔离级别性能依次递减，安全性依次递增。

  • 对于读未提交RU，V1=2 V2=2 V3=2，因为事务A在事务B还没有提交的时候就可以读到事务B修改的内容。
  • 读已提交RC，V1=1 V2=2 V3=2，因为事务A只有在事务B提交之后才会读到事务B修改的内容。
  • 可重复读RR，V1=1 V2=1 V3=2，因为在事务A中，两次读值都会和事务开启前一样。
  • 串行化，V1=1 V2=1 V3=2，和可重复读不同，这里V2=1 是因为事务B的修改是在事务A提交之后的，因为这行数据被锁住了，==注意这里是读写的情况，不是上面我们说的写写的情况，但这里还是有锁，这是因为这里说的是SQL标准的串行化，跟InnoDB没关系==。
MVCC
MVCC的概念

在介绍InnoDB的事务隔离实现之前，我们有必要先了解MVCC到底是什么。

在MySQL中，每条数据更新都会在回滚日志中记录一条回滚操作。事务在不同时刻启动，看到的值是不同的。例如当前值是4，但是在查询这条记录的时候，不同时刻启动的事务会有不同的 read-view。如图中看到的，在视图 A、B、C 里面，这一个记录的值分别是 1、2、4，同一条记录在系统中可以存在多个版本，就是数据库的多版本并发控制（MVCC）。对于 read-view A，要得到 1，就必须将当前值依次执行图中所有的回滚操作得到。

MySQL中的视图有两种，一种是用查询语句定义的虚拟表，用create view… 创建，使用和表差不多。

还有一种是InnoDB实现MVCC时用到的一致性试图，用来支持RR、RC两种事务隔离级别。

从上文可知，事务不同时刻启动能”看到”的值是不同的，并且这个实现是依赖回滚日志实现的。那具体是怎么做的呢？

MVCC多版本记录方式

对于数据库的一行数据，其存的数据不仅仅是我们定义的字段，还包含两个隐藏列：trx_id(表示修改该行事务的事务ID)；db_roll_prt(指向该行回滚段的指针)。基于db_roll_prt，在回滚日志中就组成了一个链表，且每个节点上有事务id信息。

说明白了行记录，就明白了如果寻找历史版本，但还没讲多版本是怎么选择的，也就是，事务启动之后，我们去读一行记录，到底取的是哪个版本？

这个问题的关键就是一致性视图，一致性视图是MVCC实现的关键，用来判断数据的哪个版本对当前事务可见。

MVCC确定版本的方式

事务在启动的时候会将当前正在处理的事务(事务开启了但还没提交)放入一个数组中，这个数据就是一致性视图，当在这个事务中去取某一数据行A的数据时，其过程如下：

  1. 先取当前数据行隐藏列的trx_id=12，然后和数组相对比，不在数组范围内，说明是在当前事务创建之后才有的，不可见；
  2. 然后取下一个版本trx_id=7，在数组范围内，并且在数组中找到了，说明这个trx_id=7这个事务是在当前事务创建之前就有了，但这个事务还没提交，所以也是不可见的；
  3. 然后再去trx_id=5，在数组范围内，并且没在数组中找到，这就说明这个事务是在当前事务创建之前就有了，且已经提交了，所以是可见的，最终取到的值就是trx_id=5对应的数据行；
  4. 我们假设没有trx_id=5，那么trx_id=7之后回去事务id为2的记录，这个事务比数组最小值小，说明这个事务早就有了且提交了，也是可见的。

快照读和当前读

MVCC的模式引出了快照读的概念，也就是在事务启动的时候会为当前正在处理的事务创建一个数组，然后读数据的时候会和这个数组做比对，这就是快照读大概的意思。

那么什么是当前读呢？

当前读也很好理解，就是我们读数据的时候不会走回滚日志去找以前的版本，只会取数据行当前的版本对应的数据。那什么时候会是快照读呢？

如果我们在一个事务中有update/delete语句，试想一下如果这个时候不是当前读，而是要去翻回滚日志找以前的版本，那么这个操作肯定是不准确的了，因为已经不是基于最新的数据的操作了。其实除了update/delete，一共有以下几种情况会当前读：

  1. select for update(加写锁)
  2. select lock in share mode(加读锁)
  3. update/delete
InnoDB的事务隔离

终于讲到了InnoDB的事务隔离，这里的InnoDB事务隔离说的是解决写读并发的问题(写写并发的问题InnoDB是通过锁解决的)。InnoDB默认的事务隔离级别是可重复读RR，标准的SQL隔离 RR是不能解决幻读的，但InnoDB 实现的 RR 避免了幻读问题。

InnoDB默认使用的事务隔离级别是RR，但我们也可以改成别的隔离级别，下面就分析一下InnoDB不同隔离级别的实现。

读未提交

InnoDB的读未提交是通过写锁实现的，但只保证了写写并发，也就是两个事务对同一行数据更新时，一个事务需要等待另一个事务释放写锁才能进行。但并没有保证写读的情况，所以会有脏读、不可重复读、幻读这些异常读情况。

InnoDB不同隔离级别的写写并发都是通过写锁实现的，看上节写写—写锁。

读提交和可重复读

前面说了读提交和可重复读在InnoDB中的实现都是依赖MVCC的，其实这两个的实现原理基本一样，唯一就是一致性视图生成的区别。

读提交会在每个SQL语句执行之前都重新生成一份一致性视图，这样在当前事务处理过程中又新提交的事务也是可读的，就会造成不可重复读的问题。

而可重复读只会在事务启动的时候生成一次一致性视图，以后都用这个视图，在事务的整个生命周期内可见的是事务都是相同的，也就避免了不可重复读的问题。

可重复读隔离级别下解决幻读问题

常规的SQL标准可重复读隔离级别下是不能解决幻读问题的，但InnoDB的可重复读隔离级别可以解决幻读问题，其依靠的是间隙锁+行锁(合成next key-lock)。

间隙锁就是给数据行与行之间加锁，使其不能插入数据。

串行化

串行化性能较低，但隔离性最高，所有读请求都会加上读锁，不分快照读和当前读，所有写会加上写锁

一致性

最后一个一致性，一致性是一个比较大的概念，是要保证数据的一致性，而要做到这点我们除了InnoDB事务保证了之前的原子性、隔离性、持久性之外，也需要业务系统的保证。比如一个转账操作，如果业务系统只做了转出，而没有转入，那么这个数据的一致性就被破坏了。
